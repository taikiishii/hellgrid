# HELLGRID を機械学習に自動プレイさせる — 設計と実装の解説

このドキュメントは、ブラウザFPS「HELLGRID」を強化学習(Reinforcement Learning, RL)で
自動プレイさせるまでの設計と実装を、**強化学習をやったことがない人でも読める**ように
書いたものです。要所には中級者向けのコラムを挟んでいます。

コードは `js/sim/`(シミュレーション)と `env/`(学習環境)にあります。

---

## 0. 全体像

```
                    ┌─────────────────────────────────────┐
                    │  Python (PPO / PyTorch)             │
                    │                                     │
                    │   観測 ──▶ ニューラルネット ──▶ 行動  │
                    │    ▲                          │     │
                    └────┼──────────────────────────┼─────┘
                         │  観測・報酬               │ 行動
                         │  (float32 の生バイト列)    │ (JSON)
                    ┌────┼──────────────────────────▼─────┐
                    │  Node × 12プロセス                   │
                    │                                     │
                    │   HellgridEnv × 32  (env/env.js)    │
                    │      └─ World  (js/sim/world.js)    │
                    │         描画なし・音なし・決定的        │
                    └─────────────────────────────────────┘
```

ポイントは3つです。

1. **ゲームのルール(シミュレーション)を、描画・音・入力から完全に切り離した**
2. **画面のピクセルではなく、レイキャストの結果を直接ベクトルとして観測に使う**
3. **「出口までの距離」を報酬に組み込むことで、疎報酬問題を回避した**

以下、順に説明します。

---

## 1. そもそも強化学習とは(最小限の語彙)

強化学習は「試行錯誤で上手くなる」タイプの機械学習です。登場人物は2人だけです。

| 用語 | 意味 | HELLGRID では |
|---|---|---|
| **環境** (Environment) | ルールを持った世界 | ゲームそのもの |
| **エージェント** (Agent) | 環境の中で行動する主体 | プレイヤーキャラを操るAI |
| **観測** (Observation) | エージェントが受け取る情報 | 見えている壁・敵・HP など |
| **行動** (Action) | エージェントが出す指示 | 前進・旋回・射撃・Eキー |
| **報酬** (Reward) | 行動の良し悪しを表す数値 | 敵を倒すと+、死ぬと− |
| **エピソード** (Episode) | 開始から終了までの1回のプレイ | ステージ開始〜クリア or 死亡 |
| **方策** (Policy) | 観測 → 行動 を決める関数 | ニューラルネット |

やり取りは、次の1行に尽きます。

```
観測 → [方策] → 行動 → [環境] → 次の観測 + 報酬 → [方策] → ...
```

エージェントの目的はただひとつ、**エピソード全体でもらえる報酬の合計を最大にすること**です。
「クリアしろ」とは誰も教えません。クリアしたときに大きな報酬を出すことで、
結果的にクリアするようになる、という設計をこちらがやります。

インターフェースは事実上の標準である **Gymnasium API**(旧 OpenAI Gym)に揃えます。

```python
obs = env.reset()                                  # エピソード開始
obs, reward, terminated, truncated, info = env.step(action)   # 1ステップ進める
```

`terminated` は「ゲームのルールとして終わった」(死亡・クリア)、
`truncated` は「時間切れで打ち切った」です。この2つを区別するのは大事で、
理由は後述します(→ コラム: 打ち切りと死亡は別物)。

## 2. ゲームを「環境」にするための3つの条件

既存のゲームをそのまま学習に使うことは、ほぼできません。HELLGRID も元は
`js/game.js` という2659行の一枚岩で、描画・音・入力・ゲームロジックが混ざっていました。
学習に使うには、次の3つを満たす必要があります。

### 条件1: 描画から切り離せること

学習には**数百万〜数千万ステップ**の経験が必要です。60fpsで描画しながらだと、
1000万ステップ集めるのに 10,000,000 / 60 = 46時間かかります。話になりません。

描画を捨てると HELLGRID のシミュレーションは **1コアで秒間46,000ステップ**回ります。
実に **770倍**です。ここが全体の成否を分けます。

そこで `js/game.js` を分割しました。

```
js/sim/     ← DOM も Canvas も Audio も使わない。Node でそのまま動く
js/render.js, js/sound.js, js/main.js  ← ブラウザ専用
```

`World` は「音を鳴らす」代わりに `emit('sound', 'pistol')` とイベントを投げるだけにして、
実際に鳴らすのはブラウザ側の仕事にしました。World 自身は副作用を持ちません。

### 条件2: 決定的であること(同じ入力 → 同じ結果)

元のコードは `Math.random()` を20箇所で直接呼んでいました。これだと
「さっきのバグを再現したい」も「学習を同じ条件でやり直したい」もできません。

そこで**シード付きの擬似乱数**(mulberry32)を `World` に持たせ、`Math.random()` を全廃しました。

```js
const w = new World({ seed: 42 });   // seed が同じ + 入力列が同じ → 結果は完全に同じ
```

### 条件3: 固定タイムステップであること

元のループはこうでした。

```js
const dt = (now - lastTime) / 1000;   // 実時間の経過。フレームレートで変わる
updatePlayer(dt);
```

これだと**マシンの速さでゲームの挙動が変わります**。学習環境は実時間と無関係に
動かすので、`dt` を固定値 `1/60` に固定し、ブラウザ側だけがアキュムレータで
実時間を吸収する形にしました。

```js
acc += realDt;
while (acc >= SIM_DT) { world.step(SIM_DT); acc -= SIM_DT; }   // ブラウザ側
```

> **コラム: なぜ決定性にここまでこだわるのか**
>
> 「乱数を潰しても学習には関係ないのでは?」と思うかもしれません。実際、学習そのものは
> 非決定的でも回ります。決定性が効くのは**デバッグと検証**です。
>
> 強化学習のバグは「学習が進まない」という一種類の症状にしか現れません。原因が
> 報酬設計なのか、観測のバグなのか、ハイパーパラメータなのか、環境そのもののバグなのかが
> 切り分けられない。決定的な環境があれば「seed=42 で 1000ステップ目に必ずこうなる」と
> 言えるので、二分探索でバグを追い詰められます。
>
> このリポジトリでは `tools/parity-test.py` が、**Nodeのシムとブラウザのシムが
> 同じシード・同じ入力で完全に同じ状態列を生む**ことを検証しています。これがないと、
> 「Nodeで学習した方策がブラウザで動かない」ときに原因が特定できません。

## 3. 何を「見せる」か — 観測の設計

ここが最も設計判断が問われる部分です。

### ピクセルを見せる vs 特徴量を見せる

DOOM系のAIといえば、画面をCNN(畳み込みニューラルネット)に入れる **ViZDoom** が有名です。
「人間と同じものだけを見て遊ぶ」ので絵になります。しかし:

- 描画が必要 → シミュレーションが**2〜3桁遅くなる**
- CNNの学習に大量のサンプルが必要 → **さらに桁で遅くなる**

一方 HELLGRID は**レイキャスター**です。つまり、描画とは「1列ごとにレイを飛ばして
何に当たったかを調べ、その結果を絵にする」処理です。ならば、**絵にする手前で止めて、
レイが当たったものを直接ベクトルにすればいい**。情報量はほぼ等価で、コストはゼロです。

これが本プロジェクトの一番大きな設計判断です。

### 実際の観測 `Box(1477,)`

```
┌ rays 24本 × 15ch = 360 ────────────────────────────────┐
│  [0]     壁までの距離 (0..1 に正規化)                    │
│  [1]     そのレイの先はドアか                            │
│  [2]     施錠ドアで、まだキーを持っていないか              │
│  [3]     出口スイッチか                                 │
│  [4]     そのレイ上の最も近い敵までの距離 (1 = いない)     │
│  [5-10]  敵の種類 (亡兵/散弾兵/焔鬼/牙獣/獄騎士/漂霊)     │
│  [11]    アイテムまでの距離                              │
│  [12-13] 補給品か / キーカードか                         │
│  [14]    爆発樽までの距離                               │
└────────────────────────────────────────────────────────┘
┌ grid 11×11 × 9ch = 1089 (自己中心・進行方向が上) ────────┐
│  ch0 壁   ch1 ドア   ch2 水   ch3 床の高さ               │
│  ch4 敵   ch5 アイテム・樽                               │
│  ch6 出口までのBFS距離の勾配        ← 探索の背骨          │
│  ch7 いま必要な回復までのBFS勾配     ← 10章の教訓で追加   │
│  ch8 いま必要な弾薬までのBFS勾配     ←                    │
└────────────────────────────────────────────────────────┘
┌ scalars 28 ────────────────────────────────────────────┐
│  HP / アーマー / 弾 / シェル / 武器 / 射撃クールダウン     │
│  キーカード所持 / このステージに施錠ドアがあるか            │
│  出口までの距離 / 最寄りの敵の距離と方位 / 経過時間        │
│  回復までの距離 / 弾薬までの距離 (と、そもそも在るか)      │
└────────────────────────────────────────────────────────┘
```

ch7/ch8 とスカラーの補給物資は、**最初は入れていませんでした**。そのせいで
「HPが減ったから回復を探しに行く」が原理的に不可能になり、通しクリアが破綻します。
その顛末は **10章**に書きました。**観測の設計ミスは、報酬をいくらいじっても直りません。**

**グリッドを「自己中心」にする**のが地味に効きます。プレイヤーの進行方向が常に上を
向くように回転させて切り出すので、「北向きに壁がある」ではなく「**前方に壁がある**」
という表現になります。同じ状況が同じ入力になるので、学習が速く、汎化もします。

> **コラム: 部分観測性(POMDP)というやっかいな性質**
>
> チェスや囲碁では盤面がすべて見えています(完全観測)。FPSは違います。背後の敵は
> 見えないし、さっき通った部屋に何があったかは覚えていないと分からない。こういう環境を
> **部分観測マルコフ決定過程 (POMDP)** と呼びます。
>
> 対処は主に3つ。
>
> 1. **フレームスタック**: 直近4フレームの観測を並べて入れる。速度や「さっき撃たれた」が分かる
> 2. **再帰型の方策 (LSTM/GRU)**: 過去を内部状態に畳み込む。`sb3-contrib` の `RecurrentPPO`
> 3. **観測を盛る**: 本来見えないものを観測に入れてしまう(=カンニング)
>
> 3 は強力ですが、**やりすぎると「人間には解けない情報でクリアするAI」**になります。
> 本プロジェクトでは意図的に線を引いていて、たとえば**隠し扉('*')は壁として観測させています**。
> 見た目がただの壁だからです。一方で「出口までのBFS距離」は堂々と教えています。
> 目的が「全5ステージのクリア」であって「人間らしいプレイの再現」ではないからです。
>
> **どこまで教えるかは、何を作りたいかで決まります。** ここを曖昧にしたまま観測を盛ると、
> 「学習は成功したが、何も面白くないAI」ができあがります。

> **重要: このAIはマップを「探索」していません**
>
> 誤解のないように明記しておきます。現在の設計では、エージェントは以下を毎ステップ
> 与えられています。
>
> - **出口までのBFS最短距離の勾配**(グリッドのch6)= 「どっちに行けば出口か」
> - 出口までの残り歩数、目標が出口かキーカードか(スカラー)
> - 局所グリッド11×11の中身は**視線判定なし** = 壁の裏の敵やアイテムも見えている
>
> つまりエージェントは、**探索しているのではなく、与えられた最短経路の勾配を辿っています**。
> 自力で学んでいるのは「移動」「壁を避ける」「ドアの前でEを押す」「戦闘」であって、
> **「どこに出口があるか」は学んでいません**。
>
> これは「全5ステージのクリア」というゴールに最短で到達するための、意図的な設計判断です。
> もし「自力でマップを探索するAI」を作りたいなら、観測を次のように絞る必要があります。
>
> | 方式 | 変更 | 追加で必要になるもの | 学習コスト |
> |---|---|---|---|
> | フォグ・オブ・ウォー | BFS勾配を「これまでに見たタイル」に限定、グリッドに視線判定を入れる | LSTM (`RecurrentPPO`) | 数倍 |
> | 完全に自力探索 | BFS・出口距離を全部外す | LSTM + 内発的動機付け (RND等) | 10〜100倍 |
>
> **成果を語るときは、この線引きを必ず一緒に言ってください。** 「AIがDOOMをクリアした」と
> 「AIが与えられた経路を辿ってDOOMをクリアした」では、意味がまったく違います。

## 4. 何を「させる」か — 行動の設計

```
MultiDiscrete([3, 3, 5, 3, 2, 2, 4])
                │  │  │  │  │  │  └─ 武器切替   0=なし 1=ピストル 2=ショットガン 3=ナイフ
                │  │  │  │  │  └──── E (使う)   0/1
                │  │  │  │  └─────── 射撃       0/1
                │  │  │  └────────── 上下視点   下 / なし / 上
                │  │  └───────────── 旋回       左大 / 左小 / なし / 右小 / 右大
                │  └──────────────── 左右移動   なし / 左 / 右
                └─────────────────── 前後移動   なし / 前 / 後
```

`MultiDiscrete` は「独立した複数の離散選択」です。前進しながら右に旋回しつつ撃つ、が
1つの行動で表せます。全組み合わせを1つの `Discrete(2160)` にすることもできますが、
出力層が巨大になり学習が遅くなります。

マウスの旋回は連続値ですが、**離散のバケツ5段階**に落としました。連続行動空間は
PPOでも扱えますが、離散のほうが探索が安定します。

### フレームスキップ

1つの行動を **4シムステップ(=1/15秒)** ぶん維持します。

```js
for (let i = 0; i < 4; i++) { w.look(turn/4, pitch/4); w.step(1/60); }
```

> **コラム: フレームスキップはなぜ効くのか**
>
> 60Hzで判断させると、隣り合うフレームの観測はほぼ同じです。つまり**ほとんど情報量のない
> 判断を60回/秒させて、そのぶん学習サンプルを浪費している**ことになります。
>
> フレームスキップ4にすると:
>
> - サンプル効率が4倍に(同じ経験量で4倍の実時間をカバー)
> - **時間的に信用割当(credit assignment)がしやすくなる**。「撃った」と「敵が死んだ」の
>   間のステップ数が1/4になるので、報酬の伝播が速い
> - 探索が広くなる。ランダムに選んだ行動が4フレーム持続するので、実際に前に進める
>   (60Hzでランダム行動だと、その場でブルブル震えるだけになる)
>
> DQN が Atari で使ったのもフレームスキップ4でした。経験則として 2〜8 の範囲で、
> ゲームの反応速度に合わせて選びます。FPS なら 4 が妥当です。

## 5. 報酬の設計 — ここが本丸

### 疎報酬問題

素朴に考えると、報酬は「クリアしたら +1、それ以外は 0」でいいはずです。これを
**疎報酬 (sparse reward)** と呼びます。そして、**これはほぼ確実に学習しません。**

理由: ランダムに行動して E1M1 の出口にたどり着く確率がほぼゼロだからです。
一度も報酬をもらえなければ、何を改善すればいいのか分かりません。実際、本プロジェクトの
`tools/env-test.cjs` でランダム方策を20エピソード走らせると、**クリア 0/20、死亡 10/20** です。

### 解決策: ポテンシャルベースの報酬整形

そこで「出口に近づいたらちょっと褒める」を入れます。これを **報酬整形 (reward shaping)**
と言います。ただし素朴にやると危険です。

**ここで使うのがマップ上のBFS(幅優先探索)です。** 全タイルから出口までの
「実際に歩ける最短歩数」を先に計算しておきます。

```
出口までの歩数(BFS)
  ┌──────────────────────┐
  │ ##################   │      壁は通れない
  │ #12 11 10  9  8  7#  │      水路も通れない
  │ #13 ##  ##  ##  6 #  │      段差は「登れる高さ」までしか遡らない
  │ #14 15 16  ~~  5  X  │      施錠ドアは「キーを持っていれば」通れる
  │ ####################  │
  └──────────────────────┘
```

そして報酬をこう与えます。

```
整形報酬 = (前ステップの出口までの距離 − 今の距離) × 0.1
```

1タイル近づくと +0.1、遠ざかると −0.1。**行って戻ってくると差し引きゼロ**になります。

> **コラム: なぜ「差分」でなければならないのか — Ng et al. (1999)**
>
> 「出口に近いほど毎ステップ +0.1」という設計だと何が起きるでしょうか。
> エージェントは**出口の前で永遠に足踏みします**。クリアせずに報酬を稼ぎ続けるほうが
> 得だからです。これは典型的な **報酬ハッキング (reward hacking)** です。
>
> 1999年の Ng, Harada, Russell の論文が示したのは、次の形の報酬整形なら
> **最適方策が変わらない**という定理です。
>
> ```
>   F(s, s') = γ・Φ(s') − Φ(s)
> ```
>
> Φ(s) は状態のポテンシャル(ここでは「−出口までの距離」)。この形(**ポテンシャル
> ベース報酬整形**)だと、任意の閉ループを回ったときの整形報酬の合計が必ず 0 になるので、
> 「稼ぎ続ける」ことが原理的にできません。本プロジェクトの `(前の距離 − 今の距離) × 0.1` は
> γ=1 としたこの形です。
>
> **報酬整形をするときは、必ずこの形にできないか考えてください。** 「近いほど加点」と
> 「近づいたら加点」は、字面は似ていますが挙動がまったく違います。

### 目標の自動切り替え

E1M3・E1M4・E1M5 は、**出口が施錠ドアの向こう側**にあります。BFSを張ると、出口までの
距離が「到達不能」になります。そこで、こうしています。

```
1. 出口までBFS → 届く?  → はい: 目標 = 出口
                       → いいえ: 目標 = まだ持っていないキーカード
2. キーカードを拾った瞬間に、BFSを張り直す(施錠ドアが通れるようになる)
```

キーを拾ったステップだけは整形報酬をスキップします(目標が切り替わって距離が飛ぶため)。
実際に `tools/env-test.cjs` を走らせると、こう出ます。

```
  OK  E1M1: 侵入        48歩  目標=exit
  OK  E1M2: 闘技場       14歩  目標=exit
  OK  E1M3: 処理施設      18歩  目標=key    ← 出口が施錠ドアの向こう
  OK  E1M4: 監視塔       15歩  目標=key
  OK  E1M5: 水路        19歩  目標=key
```

### 報酬の全体

| 事象 | 報酬 | 意図 |
|---|---|---|
| 敵に与えたダメージ | +0.01 / HP | 撃つこと自体を学ばせる(倒せなくても) |
| 敵を倒した | +1.0 | |
| 受けたダメージ | −0.02 / HP | 撃たれるのは損。回避を学ぶ |
| 死亡 | −10 | |
| アイテム取得 | +0.3 | |
| キーカード取得 | +3.0 | 進行の鍵なので厚めに |
| シークレット発見 | +1.0 | |
| ドアを開けた | +0.2 | Eキーを押す動機づけ |
| **ステージクリア** | **+20** | |
| 全ステージクリア | +50 | |
| 弾の消費 | −0.002 / 発 | 乱射の抑制 |
| 時間経過 | −0.002 / ステップ | もたつきの抑制 |
| **出口に近づいた** | **+0.1 / タイル** | ポテンシャル整形 |

> **コラム: 報酬設計は「バグ」を仕込む工程でもある**
>
> 強化学習エージェントは、あなたが書いた報酬関数を**文字通りに**最大化します。
> 「あなたが意図したこと」ではなく「あなたが書いたこと」を、です。有名な失敗例:
>
> - **ボートレース (CoastRunners)**: OpenAI が RL を走らせたら、コースを周回せず、
>   同じ場所でリスポーンする加点アイテムを回り続けた。得点は人間の記録を上回った
> - **お掃除ロボ**: 「ゴミを見つけたら減点」にしたら、目をつぶるようになった
>
> HELLGRID でも起こりうる罠を、実装時に潰しています。
>
> - 「敵に与えたダメージ」に上限がないと、**HPが減らない対象を延々撃つ**ことがある
>   → 死んだ敵のHPは `Math.max(0, e.hp)` でクランプして、加点が止まるようにしてある
> - 「アイテム取得」に加点すると、**敵のドロップ品を稼ぎに来る**可能性がある
>   → ゲーム側で `it.dropped` のアイテムは取得率にカウントしない(元からの仕様)
> - 時間ペナルティが強すぎると、**自殺して早くエピソードを終わらせる**のが最適になる
>   → 死亡ペナルティ(−10)を時間ペナルティの総和(3000×0.002=−6)より大きくしてある
>
> **報酬を書いたら「これを最悪の意味で悪用するとどうなるか」を必ず一度考えてください。**
> エージェントは必ずそれを見つけます。

## 6. カリキュラム学習 — いきなり本番をやらせない

全部入りでいきなり学習させても、序盤の報酬が疎すぎて立ち上がりません。段階を追います。

| 段階 | 設定 | 学ぶこと |
|---|---|---|
| 1. `nav` | 敵なし・アイテムなし・E1M1のみ | **出口に着く**。移動とドア開けだけ |
| 2. `nav-all` | 敵なし・全5ステージ | キーカードと施錠ドア。マップ汎化 |
| 3. `arena` | **敵を全滅させないと終われない** | **戦い方**。逃げると罰 |
| 4. `single` | 全部入り・1ステージ単位 | 探索と戦闘の両立 |
| 5. `campaign-mix` | 通し + 逆カリキュラム | HP・弾の持ち越し = **リソース管理** |

前の段階の重みを `--init` で引き継ぎます([tools/run-curriculum.sh](../tools/run-curriculum.sh) が全部やります)。

```bash
python env/train.py --stage nav      --steps 3000000
python env/train.py --stage nav-all  --steps 5000000 --init runs/nav/final.zip
python env/train.py --stage arena    --steps 8000000 --init runs/nav-all/final.zip
...
```

3番目が `arena`(闘技場)なのには理由があります。**最初は「E1M2 で普通にプレイする」
段階にしていましたが、まったく機能しませんでした**(キル率7.7%)。通常のステージでは
**敵を無視して走り抜けるのが本当に最適解**だからです。詳しくは10章。

5番目の `campaign-mix` は、開始ステージを {E1M1, E1M1, E1M1, E1M2, E1M3, E1M4} から
選び、E1M2 以降から始まるときだけ「消耗した状態」にします(**逆カリキュラム**)。
E1M1 スタートを高確率で混ぜているのは、破滅的忘却を防ぐためです。

> **コラム: カリキュラムは「近道」ではなく「必要条件」のことがある**
>
> カリキュラム学習は「あったほうが速い」程度に思われがちですが、疎報酬タスクでは
> **それがないと永遠に学習しない**ことがしばしばあります。
>
> 理由は探索の確率です。「敵を避けながら鍵を拾って施錠ドアを開けて出口に着く」を
> ランダム行動で一度でも達成する確率が 10⁻¹⁵ なら、何億ステップ回しても一度も
> 報酬を見ません。勾配はゼロのままです。
>
> ただし副作用もあります。**破滅的忘却 (catastrophic forgetting)**。難所だけを
> 集中訓練すると、それ以外を忘れます。本プロジェクトでは実際にこれで失敗しました
> (通し性能が 2.90 → 2.52 に悪化)。**逆カリキュラムをやるなら、簡単なケースを
> 必ず混ぜ続けること。** 顛末は10章に書きました。

## 7. アルゴリズム: なぜ PPO か

**PPO (Proximal Policy Optimization)** は、方策(=ニューラルネット)を直接勾配で
更新する手法です。ざっくり言うと:

1. 現在の方策で何万ステップか遊ぶ(**ロールアウト**)
2. 各行動が「期待より良かったか悪かったか」を計算する(**アドバンテージ**)
3. 良かった行動の確率を上げ、悪かった行動の確率を下げる
4. ただし**一度に大きく変えすぎない**ようにクリップする ← ここが "Proximal"

4 が PPO の肝です。方策が一気に変わると、それまでに集めた経験が使えなくなって
学習が発散します。「近く(proximal)にとどまれ」という制約が安定性を生みます。

> **コラム: PPO / DQN / SAC の使い分け**
>
> | | 得意 | 行動空間 | データ効率 |
> |---|---|---|---|
> | **DQN** 系 | Atari、離散で環境が安い | 離散のみ | 高(off-policy・経験再生) |
> | **PPO** | ゲーム全般、並列環境が張れる場合 | 離散/連続 両方 | 低(on-policy) |
> | **SAC** | ロボット、実機で試行回数が限られる | 連続のみ | 高(off-policy) |
>
> PPO は **on-policy** といって、「今の方策で集めたデータ」しか使えません。使い終わったら
> 捨てます。データ効率は悪いのですが、**環境が安いなら関係ありません**。並列環境を
> 192個張って毎秒7,000ステップ集められるなら、データを捨てても圧倒的に速く回ります。
>
> HELLGRID は描画を捨てたことで環境が極端に安くなったので、PPO が最適です。
> 逆に、実機ロボットのように1ステップが高価な世界では SAC のような off-policy を選びます。
>
> **「どのアルゴリズムか」より「環境のステップが1つ何マイクロ秒か」のほうが、
> 学習が回るかどうかを決めます。** これが本プロジェクトでフェーズ1(描画の分離)に
> 全体の8割の労力を割いた理由です。

## 8. 並列化とスループット — 実測値

Node は**シングルスレッド**です。1プロセスに環境を何個並べても、コアは1つしか使えません。

```
1コアのシミュレーション本体      46,000 step/秒   (描画なし)
  + 観測の生成                   3,500 行動/秒   (フレームスキップ4 = 14,000 step/秒)
  + Python との通信 (1プロセス)   2,350 行動/秒
  × 12プロセス並列               8,800 行動/秒   (35,000 sim step/秒)
  + PPO の学習(方策の順伝播・逆伝播)  3,500〜7,300 FPS
```

対策は「Nodeプロセスを複数立ち上げ、環境を分割して持たせる」ことです。
`env/hellgrid_env.py` の `HellgridVecEnv` がそれをやります。

```python
venv = HellgridVecEnv(num_envs=192, n_workers=12)   # 12プロセス × 16環境
```

通信は stdin/stdout。行動はJSON、観測は **float32 の生バイト列**で返します。

> **コラム: 実際に踏んだ性能バグ**
>
> 最初の実装では、8ワーカーに増やしても **3,600 行動/秒**しか出ませんでした
> (1ワーカーが2,350なので、8倍の18,800が出るはず)。原因は2つ。
>
> 1. **毎ステップ、全環境ぶんの `info` 辞書をJSONにしていた**
>    12個のフィールドを持つ辞書を、192環境ぶん、毎ステップ。JSONのシリアライズだけで
>    スループットが3割落ちていました。→ **エピソードが終わった環境のぶんだけ送る**ように変更。
>
> 2. **毎ステップ観測バッファを新規確保していた**
>    `Float32Array.from(obs)` を192個 × 秒間数千回 = GCが悲鳴を上げます。
>    → 送信バッファを1つ確保して使い回すように変更。
>
> 直した結果、8,800 行動/秒(2.4倍)。**RLの性能問題は、たいていアルゴリズムではなく
> 「環境とのやりとり」にあります。** プロファイルを取らずにハイパーパラメータを
> いじり始めるのが典型的な時間の溶かし方です。

## 9. デバッグの実践

強化学習のバグは「学習が進まない」という**たった一種類の症状**にしか現れません。
だからこそ、学習を回す前に切り分けておく必要があります。本プロジェクトでやっていること:

### (1) ランダム方策のベースラインを取る

`tools/env-test.cjs` はランダム行動で20エピソード走らせて、こう出します。

```
ランダム方策 20エピソード:
  死亡 10/20   クリア 0/20
  報酬 中央値 -2.2  最小 -12.3  最大 -0.3
```

**クリアが 0/20 であることが重要です。** もしランダムでクリアできてしまうなら、
タスクが簡単すぎるか、報酬にバグがあります。逆に学習後の方策がこれを上回らないなら、
学習が機能していません。

### (2) 観測の値域を機械的にチェックする

```
観測の健全性: NaN/Inf 0個  値域 [-1.000, 1.000] OK
```

NaN が1つでも混じると、ニューラルネットの重みが全部 NaN になって静かに死にます。
値域が [-1, 1] に収まっているかも見ます(スケールがバラバラだと学習が遅くなります)。

### (3) 報酬整形の符号を単体テストする

「出口に向かって1歩進んだら、必ず正の報酬になる」ことをテストしています。

```
ポテンシャル整形: 出口へ47歩ぶん進めて全ステップ正の報酬  (出口に到達) OK
  開始時の出口までの距離 48歩 -> 1歩
```

BFSの向き(段差の登り/降りの非対称性)を間違えると符号が逆になりますが、
学習を回してからでは絶対に気づけません。

> **コラム: 実際に踏んだ「見えないバグ」**
>
> 学習を回したら、TensorBoard の `clear_rate` がずっと 0 のままでした。しかし同時に
> `ep_rew_mean` は −2.5 → +24 に上がり、`ep_len_mean` は 1500 → 180 に激減している。
> つまり**明らかにクリアしているのに、クリア率が0と表示されていた**。
>
> 原因は SB3 の `VecMonitor` でした。これは `info["episode"]` に `{r, l, t}` を
> 書き込むラッパーで、こちらが入れておいた `info["episode"]["cleared"]` を
> **丸ごと上書きしていた**のです。
>
> ```python
> # 修正前: VecMonitor に上書きされて常に 0
> self.cleared.append(1 if ep.get("cleared", 0) > 0 else 0)
> # 修正後: Node から来た生のフィールドを直接見る
> self.cleared.append(1 if info.get("levelsCleared", 0) > 0 else 0)
> ```
>
> **教訓: 指標は必ず2系統以上見ること。** `clear_rate` だけを見ていたら
> 「学習が失敗している」と誤診して、報酬設計をいじり始めていたはずです。
> `ep_len_mean` が急落したことが、真実を教えてくれました。

## 10. 到達点と、通しクリアで壁にぶつかった話

ここからは**うまくいかなかった話**です。教科書には載らないが、実際にやると必ず出会う
たぐいの失敗なので、診断の過程ごと記録しておきます。

### うまくいったところ

```
stage=single (全5ステージ・敵もアイテムもあり・1ステージ単位)

  クリア率   100.0%  (100/100)     全5ステージそれぞれ 100%
  平均所要時間 9.1秒
```

**1ステージ単位なら、全5ステージを100%クリアします。** キーカードが要る
E1M3〜M5 も、鍵を拾ってから出口へ向かう2段階の行動をこなします。

### 壁: 通し(campaign)は平均2.9ステージで力尽きる

HPと弾がステージをまたいで持ち越される「通し」にすると、こうなりました。

```
到達ステージの分布 (40エピソード)      死亡率 100%
  2ステージ   5  #####
  3ステージ  34  ##################   ← E1M4 (監視塔) で全滅
  4ステージ   1  #
  5ステージ   0                        ← 完走ゼロ
```

原因を3段階で突き止めたので、順に書きます。

---

### 診断1: 割引率 γ が短すぎた

最初の実装は γ=0.99 でした。**γ の実効的な視野は約 1/(1−γ) ステップ**なので、100ステップ先までしか見えません。

| 見ようとしているもの | 距離 | 割引後の価値 |
|---|---|---|
| 次のステージのクリア (+20) | 約150ステップ先 | 20 × 0.99¹⁵⁰ = **+4.4** |
| 全ステージ完走 (+50) | 約600ステップ先 | 50 × 0.99⁶⁰⁰ = **+0.12** |

**エージェントには「4ステージ先まで生き延びる価値」が事実上見えていませんでした。**
目先の1ステージだけを見て走るので、`single` とほぼ同じ振る舞いになり、HPが尽きて死ぬ。

> **コラム: γ はハイパーパラメータではなく、問題定義の一部です**
>
> γ を「学習率みたいな微調整パラメータ」だと思っていると、この罠に落ちます。
> γ は **「エージェントに何を目的だと思わせるか」** を決めています。
>
> γ=0.99 のエージェントは、**「100ステップ先までの利益を最大化せよ」という、
> あなたが出したのとは違う問題を解いています。** それは間違った答えではなく、
> **違う問いへの正しい答え**です。
>
> エピソード長が変わったら γ も見直す。これは調整ではなく、**問題を定義し直す**作業です。

γ=0.999(視野1000ステップ)にすると、頭打ち(2.4で停滞)と後期の劣化が両方消えました。

---

### 診断2: 報酬が非対称で、回復する理由が存在しなかった

報酬表を見返して気づきました。

| | 重み | HP 1あたり |
|---|---|---|
| 被ダメージ | −0.02 / HP | **−0.020** |
| メディキット (+25 HP) | +0.3(一律) | **+0.012** |

**回復の価値が、ダメージのコストの6割しかありません。** HPそのものには報酬がなく、
「死なない」ことを通じてしか価値を持たない。そしてその「死」の先にある報酬は、
診断1のとおり割引で消えていた。

**結果、回復アイテムを拾う理由がエージェントには構造的に存在しませんでした。**
取得率0%は、バグではなく**設計どおり**だったのです。

修正: HP+アーマーの増減を**対称に**評価する(±0.02/HP)。取得率は 0% → 10% に。

---

### 診断3: 介入実験で「慎重さ」の欠如を測る

ここで、推測をやめて**測る**ことにしました。

**実際のプレイから1500状態を採取し、観測のHPスカラー1個だけを書き換えて**
方策に通します。敵の位置も壁も弾も出口までの距離も完全に同一なので、
行動が変われば原因はHPだと言い切れます([tools/ablate-hp.py](../tools/ablate-hp.py))。

```
HP=100 のときと比べて、行動が変わった状態の割合
  HP= 10   12.6%
  HP= 25   10.7%
  HP= 50    7.3%
  HP= 75    4.0%
```

HPは確かに効いています。**が、中身を見ると絶望的でした。**

| HP | 前進率 | **後退率** | **射撃率** |
|---|---|---|---|
| 10(瀕死) | 87.4% | **3.0%** | **43.3%** |
| 100(万全) | 88.9% | **2.5%** | **44.5%** |

**瀕死でも87%の確率で前進し続け、後退率は2.5%→3.0%にしか増えません。**
慎重さの概念を獲得していない。これが E1M4 で100%死ぬ直接の原因でした。

さらに副産物として、**射撃率43%なのにキル率0.2%**という事実が出ました。
**常時トリガーを引きながら、何にも当てていない。** 弾を撒き散らして走っているだけです。

> **コラム: アブレーションは「なぜ動くか」を測る唯一の実用的手段**
>
> ニューラルネットの中身は読めません。しかし**入力を1つだけ書き換えて出力の変化を見る**
> ことはできます。これは因果推論でいう介入(intervention)そのもので、相関ではなく
> **原因**を特定できます。
>
> RLのデバッグでは、報酬曲線を眺めるより、この手の介入実験のほうがはるかに情報量が
> 多いことがよくあります。**「学習が進まない」ではなく「何を学習していないか」**を
> 特定できるからです。

---

### 診断4: 逆カリキュラムは失敗した(そして、なぜ失敗したか)

「難所(E1M4)を、実戦と同じ消耗した状態から集中訓練すれば慎重さを学ぶだろう」
という仮説を立て、E1M3/E1M4 から HP 20〜90 で始まる訓練を15Mステップ回しました。

**外れました。** しかも本番性能が悪化しました。

| | 逆カリキュラム前 | 後 |
|---|---|---|
| 通しの平均到達ステージ | **2.90** | **2.52** ← 悪化 |
| 瀕死時の前進率 | 87.4% | **90.9%** ← より無謀に |
| 射撃率 | 43% | **60%** ← ばら撒きが悪化 |

原因は**破滅的忘却(catastrophic forgetting)**でした。E1M3/E1M4 からしか始めないので、
**E1M1・E1M2 を練習しない15Mステップの間に、序盤の走り方を忘れた**のです。

> **コラム: 破滅的忘却は「難所だけ練習する」の必然的な代償**
>
> 人間なら「苦手なところだけ練習する」は正しい戦略です。しかしニューラルネットには
> 「前に覚えたことを保護する」機構がありません。**勾配降下は、いま見ているデータに
> 対して最適な重みへ、容赦なく引っぱります。**
>
> 逆カリキュラムをやるなら、**必ず簡単なケースを混ぜ続ける**こと。今回の修正では、
> 開始ステージを {E1M1, E1M1, E1M1, E1M2, E1M3, E1M4} から選び、E1M1 を必ず
> 高確率で混ぜるようにしました(E1M1 スタートのときは消耗させない)。

---

### 診断5: 根本原因 — 生き延びる手段が3つとも欠けていた

ここまでの実測をまとめると、エージェントには生存の手段が**3つとも**ありませんでした。

| 手段 | なぜできないか |
|---|---|
| **逃げる** | 慎重さを学んでいない(瀕死でも90%前進) |
| **倒す** | 撃つことと当てることが結びついていない(射撃60%・キル0.2%) |
| **回復する** | **視界外の回復アイテムを探す手段が観測に存在しない** |

3つめが決定的でした。BFS距離場は**「出口まで」しか作っていなかった**のです。
アイテムはレイ(視界内)か11×11グリッド(±5タイル)に入って初めて見える。

**つまり「HPが減ったから回復を探しに行こう」という行動は、いまの観測では
原理的に実行不可能でした。** できるのは、たまたま通り道にあったものを拾うことだけ。

**手段がないものは、いくら報酬を与えても学習されません。** これは強化学習の
最も基本的な制約で、報酬設計をいくらいじっても解決しません。

### 修正: 観測を作り直す

出口と同じ仕組みで、**「いま自分に必要な補給物資」への BFS 勾配**を追加しました。

```
grid ch6 … 出口 (または必要なキーカード) への勾配   ← 元からあった
grid ch7 … いま必要な回復・アーマーへの勾配        ← 追加
grid ch8 … いま必要な弾薬への勾配                 ← 追加
scalars  … 回復まで / 弾薬まで の残り歩数と、到達可能か  ← 追加
```

満タンなら種がゼロになり、場そのものが消えます(=「探す先がない」)。
毎ステップ BFS を張り直すと重いので、**「拾った / 満タンになった / キーを取った」
ときだけ**張り直します([env/obs.js](../env/obs.js) の `supplySignature`)。

観測は `Box(1231,)` → **`Box(1477,)`** になりました。

あわせて、戦闘を学ばせるための **`arena` モード**を作りました。
**敵を全滅させないと終われず、出口から逃げると罰**を受けます。

> **コラム: 「戦闘を学ばない」のは方策のせいではなく、環境のせい**
>
> 通常のステージでは「敵を無視して出口へ走る」が**本当に最適解**です。移動速度は
> プレイヤー 4.2 タイル/秒に対し、最速の敵でも 3.6。**全部振り切れる。**
>
> この状況で「戦闘を学べ」と願っても無駄です。エージェントは正しく報酬を最大化して
> いるだけ。**戦闘を学ばせたいなら、戦闘しか出口がない状況に追い込むしかない。**
> `combat` 段階(E1M2で普通にプレイ)がキル率7.7%で終わったのは、そこを取り違えて
> いたからです。

### 教訓

1. **γ はエピソード長に合わせて設計する。** ハイパーパラメータではなく問題定義の一部
2. **報酬の対称性を疑う。** 「損は罰するが得は褒めない」設計は、必ず歪んだ方策を生む
3. **測れ。** アブレーション(入力を1つだけ変える)は、報酬曲線より雄弁
4. **逆カリキュラムには簡単なケースを混ぜる。** さもないと忘れる
5. **手段がないものは学習されない。** 報酬設計をいじる前に、**その行動が観測から
   実行可能か**を確認する

**5番が一番痛い教訓でした。** 「回復を拾わないのは報酬が足りないからだ」と思って
報酬を2回いじりましたが、本当の原因は**回復の場所を知る手段がなかった**ことでした。

## 付録: 用語集

| 用語 | 意味 |
|---|---|
| **エピソード** | 開始から終了(死亡・クリア・時間切れ)までの1回のプレイ |
| **ロールアウト** | 現在の方策で環境を回して経験を集めること |
| **on-policy / off-policy** | 今の方策で集めたデータしか使えない / 過去のデータも再利用できる |
| **アドバンテージ** | その行動が「平均より」どれだけ良かったか |
| **疎報酬 (sparse reward)** | クリア時にしか報酬が出ない状況。そのままではまず学習しない |
| **報酬整形 (reward shaping)** | 途中経過に報酬を出して学習を助けること |
| **ポテンシャルベース報酬整形** | γΦ(s')−Φ(s) の形。最適方策を変えないことが証明されている |
| **報酬ハッキング** | 意図しない方法で報酬を最大化すること |
| **POMDP** | 部分観測。世界の一部しか見えない環境 |
| **フレームスキップ** | 1つの行動をNフレーム維持すること |
| **破滅的忘却** | 新しいタスクを学ぶうちに前のタスクを忘れること |
| **VecEnv** | 複数環境をまとめて `step()` するインターフェース |

> **コラム: 打ち切りと死亡は別物**
>
> Gymnasium が `terminated`(ゲームのルールとして終了)と `truncated`(時間切れ)を
> 分けているのには、数学的な理由があります。
>
> 価値関数 V(s) は「この状態から先、将来もらえる報酬の合計の期待値」です。
> 死亡なら V(終端) = 0 で正しい。しかし**時間切れは環境の都合で打ち切っただけで、
> ゲーム自体はまだ続いていた**わけです。ここで V = 0 としてしまうと、
> 「時間切れ間際の状態は価値がない」という嘘を学習してしまいます。
>
> 正しくは、時間切れの場合は打ち切り時点の観測から V を推定して**ブートストラップ**します。
> SB3 では `info["TimeLimit.truncated"]` を見てこれを自動でやってくれます。
> だから `env/protocol.py` はこのフラグをちゃんと立てています。
>
> **自作環境でここを間違えると、学習が「なんとなく遅い」という形でだけ現れます。**
> よくある落とし穴です。
