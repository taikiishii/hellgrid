# 汎化への計画 — 「5ステージ最適化」から「任意ステージ対応」へ

## 1. 診断: いま何に最適化されているか

- **観測は最初から汎化する設計**。obs2.js は「見たものだけ」の局所視界＋記憶した地図で、
  目標/回復/弾薬への距離場を**既知マップ上のBFS**で毎回計算する。鍵(keyTiles・seenRed/Blue)、
  施錠ドア(R/B＋鍵ロジック)、ドア(D)、アイテム(h/H/a/A/s/S/p/V)、ゲート出口を**汎用的に符号化
  済み**。ステージ固有の座標を埋め込んでいない。→ **観測の作り直しは不要。**
- **ボトルネックは学習データの多様性**。手続き迷路 (mazegen.js) が生成するのは
  `# . P X` と敵 `Z/G/I` の**わずか7種だけ**。実ステージが使う豊富な要素——
  鍵・施錠ドア・ドア・全アイテム・リフト・水路・動く歩道・爆発樽・隠し扉・
  アンブッシュ・高低差・テーマ配色・敵種 M/K/F——を**一切生成しない**。
- 通し/ハンター系 (camp-hunt*) は `mazeMix≈0` で**固定5ステージだけ**に最適化した。
  結果、鍵・敵・出口の位置を**丸暗記**し、汎用スキルではなくレイアウト記憶で解いている
  疑いが濃い(迷路系の系譜を通っているので幾何の汎化はある程度あるが、
  鍵・ドア・資源管理を伴う「実ステージ級の未知構造」は訓練で一度も経験していない)。

**結論**: 汎化は「観測の問題」ではなく「訓練データの問題」。**手続き生成器を実ステージ級の
フル機能に拡張し、そこで学習する**のが本筋。sim も観測もすでに全要素を扱えるので、
生成器が妥当なタイルマップを吐けば残りはタダで付いてくる。

## 2. 方針: フル機能の手続きステージ生成器 + ドメインランダム化

`mazegen.js` を「ステージ生成器」に拡張し、**エピソードごとにパラメータを広く抽選**して
二度と同じステージが出ないようにする(ドメインrandomization)。難易度もテーマもばらつかせる。

### 2.1 生成器に足す要素(優先順)

| 優先 | 要素 | 内容 | 汎化上の意味 |
|---|---|---|---|
| ★★★ | 鍵＋施錠ドア | r/b 鍵と R/B ドアを配置。鍵→ドア→出口の依存を保証(0〜2段) | 実ステージ最大の未学習課題(三重課題)。E1M3/M4 が固定ステージでしか出ない |
| ★★★ | アイテム | h/H/a/A/s/S/p/V を密度可変で配置(資源潤沢〜枯渇) | 回復/弾薬管理の汎化。healSeek/ammoSeek 整形が活きる |
| ★★★ | 敵の多様化 | Z/G/I に M(牙獣)/K(獄騎士)/F(漂霊) を追加、混合比・密度・休眠/トリガーを可変 | 戦闘の汎化。今は3種のみ |
| ★★ | 構造 | 部屋/廊下/braid に加え、リフト L・高低差 heights・水路 ~・動く歩道・爆発樽 o・隠し扉 * | 立体・地形の汎化 |
| ★★ | テーマ | ceil/floor/fog 配色＋壁テクスチャ混合(#/&/=)をパラメータ束で | 見た目のばらつきに対する頑健性 |
| ★ | アンブッシュ | T トリガー＋休眠敵で「進むと湧く」状況 | 記憶と警戒の汎化 |

### 2.2 難易度・テーマのパラメータ(エピソード抽選)

- **サイズ** (11〜31)、**部屋率/braid** (開けた⇔迷路的)
- **敵密度** (0〜高) と **敵種ミックス** (弱編成⇔火力/機動/タンク混成)
- **資源スカーシティ** (回復・弾の量: 潤沢⇔枯渇)
- **鍵ロック深度** (0=なし / 1 / 2段)
- **立体度** (フラット⇔多段) と **ハザード率** (水路/樽)
- **テーマ**: 上記を束ねたプリセット(例「密迷路」「開放アリーナ」「鍵ハント」「資源枯渇」
  「立体要塞」)＋配色。プリセット選択→各パラメータをその範囲で微randomize。

## 3. 学習レジメン

- **主教材を生成ステージに置く**(無限の多様性で汎用スキルを強制)。固定5ステージは
  混ぜても少量、または**混ぜない**。
- **難易度カリキュラム**は「生成パラメータ」に対して行う(易→難)。特定ステージではなく
  分布を進める。既存の逆カリキュラム(消耗スタート)も生成ステージ上で同様に。
- **既存の武器はそのまま流用**: fog観測・frontier/heal/ammo整形・killGate/gun-kill-gate は
  すべてレイアウト非依存。生成ステージにそのまま乗る。
- 生成ステージを**チェーンして「生成通し」**を作れば、通し完走の汎化版になる。

## 4. 汎化の評価(最重要の規律)

- **ホールドアウト生成ステージ**(学習と重ならないシード帯)を難易度/テーマ別バケツで評価し、
  **性能の分布**を見る(平均だけでなく、どの難易度/テーマで崩れるか)。
- **固定5ステージを「未知テスト集合」に降格**。学習に使わず、そこでの完走率を
  「真の汎化」の指標にする(手作りの良質設計に転移できるか)。
- **アブレーション**: 鍵なし/資源枯渇/立体/敵種別など、特徴を落として弱点を可視化。
- 「計測プロトコルの罠」(横断教訓2)を踏襲し、新旧は同一の生成シード帯・開始条件で比較。

## 5. 段階的ロードマップ

1. **Phase 1 — 鍵・ドア・アイテムの生成**(最大のギャップ)。生成器に r/b/R/B/D と
   アイテムを追加し、**鍵→ドア→出口の到達可能性を保証する solvability チェック**を実装
   (現状の「出口到達可能」BFS を鍵ロジック込みに拡張)。まずナビ(走り抜け)で学習し、
   ホールドアウトと実5面で汎化を測る。
2. **Phase 2 — テーマ・立体・ハザード**。配色/テクスチャ、heights、水路/樽/リフトを追加して学習。
3. **Phase 3 — 戦闘の汎化**。敵種 M/K/F と密度/ミックスを広げ、gun-kill-gate を生成ステージに乗せる。
4. **Phase 4 — 生成通し**。生成ステージを N 枚チェーンした通しで、汎化した戦闘ハンターを仕上げる。

各 Phase は既存の実験ノート様式(問題→仮説→試行→結果→考察)で
[experiment-journal.md](experiment-journal.md) に記録する。

## 6. リスクと注意

- **可解性の保証が要**。生成ステージは「鍵が取れて・ドアが開けて・出口に届く」ことを
  生成時にBFSで検証し、満たさないものは捨てる(捨てた率をログ)。戦闘ゲート付きなら
  「必要な弾/回復が取れる」量も担保する。**解けないステージは学習を殺す**(v1 arena の教訓)。
- **公平性**。資源枯渇×高密度×2段ロックのような理不尽な組合せは、可解範囲内に抑える。
  難易度は各軸を独立に振るのでなく、プリセットで整合を取る。
- **手作り設計 vs 生成**。実5面は良質な手作り。生成ステージはランダムで雑になりうるが、
  汎化学習にはそれでよい。**磨いたデモ用には実5面を残す**(用途を分ける)。
- **計算量**。生成は毎エピソード走るが、mazegen は既に高速(決定的シード)。可解性BFSの
  追加コストは小さい。学習の律速は従来どおり勾配更新。

## 7. まとめ

観測は既に汎化設計。**足りないのは「実ステージ級の多様な訓練データ」だけ**。生成器を
フル機能化し、難易度・テーマを広く振ってドメインランダム化すれば、既存の学習レシピ
(fog観測・整形・ゲート)をそのまま使って汎化できる。固定5面はテスト集合＋磨いたデモに回す。
まず Phase 1(鍵・ドア・アイテム生成＋可解性保証)から着手するのが費用対効果が高い。
