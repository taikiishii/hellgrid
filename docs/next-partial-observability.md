# 次のプロジェクト: 「見たものだけ」で学習するエージェント

現在のエージェントは**カンニングしています**。人間のプレイヤーと同じ条件、つまり
**これまでに実際に見た情報だけ**でプレイを学習させる、というのが次の目標です。

このドキュメントは、新しいセッションでこのプロジェクトを始めるための設計書です。
現状の到達点は [machine-learning.md](machine-learning.md) を先に読んでください。

## 1. いま何がチートなのか — 棚卸し

観測 `Box(1477,)` の内訳を、**正当 / チート**で仕分けします。

| ブロック | 次元 | 判定 | 理由 |
|---|---|---|---|
| **rays 24本 × 15ch** | 360 | ✅ **正当** | `hasLineOfSight()` で視線判定している。壁の裏は見えない |
| **grid ch0-5**(壁・ドア・水・高さ・敵・アイテム) | 726 | ❌ **透視** | **視線判定なし。±5タイル以内なら壁の裏も見える** |
| **grid ch6**(出口へのBFS勾配) | 121 | ❌ **地図オラクル** | マップ全体を知らないと計算できない |
| **grid ch7/ch8**(回復・弾薬へのBFS勾配) | 242 | ❌ **地図オラクル** | 同上。「見たことのない回復」の方向まで教えている |
| **scalars 0-11**(HP・弾・武器・キー所持) | 12 | ✅ **正当** | HUDに出ている情報 |
| **scalars 12-13**(このステージに赤/青ドアがあるか) | 2 | ❌ **事前知識** | 見る前から知っている |
| **scalars 14-16**(出口までの歩数・到達可能か・目標がキーか) | 3 | ❌ **地図オラクル** | |
| **scalars 17-18**(キル率・アイテム取得率) | 2 | ✅ **正当** | HUDに出ている |
| **scalars 19-20**(高さ・経過時間) | 2 | ✅ **正当** | |
| **scalars 21-23**(最寄りの敵の距離と方位) | 3 | ✅ **正当** | 視線判定あり |
| **scalars 24-27**(回復・弾薬までの歩数) | 4 | ❌ **地図オラクル** | |

**チートの合計 1096次元 / 1477次元 = 74%。** 特に **grid の ch6/ch7/ch8(BFS勾配)が
探索の背骨**になっており、これを外すと**エージェントは出口の場所を知る手段を完全に失います。**

> 現状のエージェントが「マップを探索していない」のはこのためです。
> **探索しているのではなく、与えられた最短経路の勾配を辿っているだけ**です。

## 2. 設計の核心 — 「記憶」をどう与えるか

観測を削るだけでは学習しません。**代わりに記憶を持たせる**必要があります。
ここが本プロジェクトの本質的な設計判断です。

### 難易度の3段階

| | 与えるもの | 必要な技術 | 学習コスト | 「人間らしさ」 |
|---|---|---|---|---|
| **Lv1** | 自分が見たタイルだけの地図(フォグ・オブ・ウォー)+ **その地図上での**BFS | LSTM不要 | 現状の2〜3倍 | 中 |
| **Lv2** | 自分が見たタイルだけの地図。**BFSなし** | RecurrentPPO (LSTM) | 現状の10〜30倍 | **高** |
| **Lv3** | 地図なし。レイとスカラーだけ | LSTM + 内発的動機付け (RND) | 現状の50〜200倍 | 最高 |

**推奨は Lv2 を目標に、Lv1 を踏み台にすること。**

Lv1 の「見た地図上でのBFS」は**カンニングではありません**。人間も、探索済みのエリアなら
「あっちに行けば元の部屋に戻れる」と分かります。**未探索の領域は BFS が届かない**ので、
「どこへ行けば新しいものが見えるか」は自力で学ぶ必要があります。

Lv3 は研究レベルです。**5ステージのクリアは保証できません。** 最初から狙わないこと。

### フォグ・オブ・ウォーの実装方針

環境側(`env/`)に**エージェント自身の記憶**を持たせます。**エージェントが実際に見た
情報しか入れない**ので、これはカンニングではなく「メモを取ること」に相当します。

```
毎ステップ:
  1. プレイヤーの視界(FOV内 + 視線が通るタイル)を計算  ← World.hasLineOfSight を再利用
  2. 見えたタイルを known[] にマーク。壁・ドア・水・高さを記録
  3. 見えた敵・アイテムを「最後に見た位置と時刻」として記録 (時間で減衰)
  4. known[] だけを観測に出す
```

**重要**: 敵は動くので「さっき見た場所にもういない」ことがあります。**最後に見た位置 +
経過時間**を渡し、古い情報は信頼できないことをエージェントに学ばせます。これは人間の
プレイと同じです。

### 観測の設計案 (Lv2)

```
┌ rays 24本 × 15ch = 360 ──────────────────────── そのまま (元から正当)
├ 局所マップ 15×15 × 8ch = 1800 ──── 自己中心・進行方向が上。known[] のみ
│   ch0 未探索 (1 = まだ見ていない)      ← これが探索の主役になる
│   ch1 壁      ch2 床      ch3 ドア (施錠状態込み)
│   ch4 水・危険                        ch5 床の高さ
│   ch6 最後に見た敵 (経過時間で減衰)   ch7 最後に見たアイテム (種別は別チャネル案も)
├ 全体マップ 24×24 × 4ch = 2304 ──── レベル全体を粗く。絶対座標・自機マーカー付き
│   ch0 未探索   ch1 壁   ch2 見たアイテム   ch3 自分の軌跡 (訪問回数)
└ scalars ~20 ─────────────────────── HP・弾・武器・キー・探索率・最後に新タイルを
                                       見てからの経過ステップ 等
```

**全体マップを入れるかどうかが最大の判断です。** 入れれば「さっき来た道」を覚えられるので
LSTM の負担が激減し、学習が現実的になります。入れなければ LSTM が全部覚える必要があり、
研究レベルの難しさになります。**最初は入れることを強く推奨します。**

## 3. 報酬の設計

**BFSポテンシャル整形は消えます。** これがナビゲーションの背骨だったので、代わりが要ります。

| 報酬 | 値 | 意図 |
|---|---|---|
| **新しいタイルを見た** | **+0.02 / タイル** | **探索の主動力。これが新しい背骨になる** |
| 出口スイッチを初めて見た | +5.0 | 「見つけた」ことを強く褒める |
| 出口を見つけた後の接近 | +0.1 / タイル | **見えている出口へのBFS**なら正当 |
| キーカードを見つけた | +2.0 | |
| ステージクリア | +20 | |
| その他(HP・キル・アイテム・時間) | 現状のまま | |
| **同じ場所をうろつく** | −0.01 / ステップ | 訪問回数に応じたペナルティ |

> **注意**: 「新しいタイルを見た」報酬は、**部屋の隅を舐めるだけの方策**を生むことがあります
> (探索報酬ハッキング)。時間ペナルティとのバランスを必ず実測すること。
> 心配なら **RND (Random Network Distillation)** に置き換える。

## 4. アルゴリズムと学習

- **RecurrentPPO** (`sb3-contrib`)。LSTM 256ユニット程度から
- **γ = 0.999**(エピソードが長くなるので必須)
- **手続き生成マップ**での学習を強く推奨。`tools/gen-*.js` を拡張して、
  ランダムな迷路 + アイテム配置を無限に作る。**5マップだけで学習すると丸暗記します**
  (現状は BFS オラクルのおかげで丸暗記が問題にならなかったが、探索版では致命的)
- **カリキュラム**:
  1. 小さいランダム迷路(10×10、敵なし)で「探索して出口を見つける」
  2. マップを大きく(20×20 → 30×30)
  3. 敵とアイテムを戻す
  4. E1M1〜E1M5 に転移
  5. 通し

## 5. 計算コスト — 実測に基づく見積もり

**これは現状の10〜30倍の学習コストがかかり、成功が保証されません。** 正直に見積もります。

| | 現状 | 探索版 (Lv2) の見込み |
|---|---|---|
| 学習ステップ | 53M(約5時間) | **500M〜1.5G** |
| E1M1 クリア | 100% | 60〜90%(楽観的に) |
| 5ステージ通し完走 | 76.7% | **10〜40%(かなり楽観的に)** |

### 律速はどこか — 測ったら、思い込みと逆だった

「シミュレーションが重いから環境が律速」と長く思い込んでいました。**測ったら逆でした。**

```
192環境 / 12ワーカー / Core Ultra 7 165U (15Wのノート用CPU、12コア)

  環境だけ (IPC込み)     12,692 行動/秒
  学習器だけ (CPU)        2,628 行動/秒   ← こちらが遅い
  ─────────────────────────────────────
  時間の内訳   環境 17%   学習器 83%
```

env.step の中身も測りました。

```
  シムだけ         61,538 step/秒  ( 16 マイクロ秒/step)
  観測の生成だけ    4,838 回/秒    (207 マイクロ秒/回)   ← env内では観測が83%

  env.step の内訳:  シム4回 65us (26%) / 観測の生成 207us (83%)
```

**つまり: 環境の中では観測の生成が重いが、学習ループ全体では学習器(PyTorch on CPU)が83%。**

この2つを混同すると、**最適化の努力を全部むだにします**:

- ❌ **観測の生成を最適化する** → 全体の17% の、そのまた83% = **せいぜい14%しか効かない**
- ✅ **学習器を速くする(GPU / 網を縮める)** → **全体の83%に効く**

### 学習器のスループット (CPU実測)

| 構成 | 行動/秒 |
|---|---|
| 現行 obs1477 → 512×512 (MLP) | 8,936 |
| **網を縮める** obs1477 → 256×256 | **17,410**(**約2倍**) |
| 次期 obs4100 → 512×512 + LSTM256 | 2,716 |
| 次期 obs4100 → **256×256** + LSTM256 | **4,456** |
| 次期 obs2000(観測を絞る) → 256×256 + LSTM256 | **5,359** |

**まず試すべきは「網を縮める」。** タダで約2倍です。
観測を 4100 → 2000 に絞るのも効きます(全体マップを 24×24 → 16×16 に、など)。

1. **環境が Node.js です。** Colab は Python ノートブック。`apt install nodejs` して

   500M ステップ = 29時間 → **最低2セッション**。**Google Drive へのチェックポイント保存と

3. **バックグラウンド実行は Pro+ ($49.99) だけ**。Pro ではブラウザを閉じると止まります
4. **GPU上の学習器スループットは外挿値です。** 課金する前に、**10分の実測**をしてください
   (Colab で `!nproc` と、この設計書のベンチを1回回すだけ)

### 最初にやるべきこと(タダの高速化)

課金する前に、**手元で以下を試すこと**。

1. **`net_arch` を 512×512 → 256×256 に**。学習器が約2倍(実測)

### 学習結果: カリキュラム第1段階 maze9 (9×9 迷路)

パイプライン接続済み (`server.js` の `cfg.env2` / `train2.py` / `eval2.py`)。
PPO (MLP 256×256, γ=0.995, LSTM なし) を 3M 步、約40分 (CPU, 96環境)。

```
評価 100 エピソード (学習に出ていないシードの迷路 = 丸暗記ではない):
  クリア率        92.0%   (ランダム方策の基準値は ~5%)
  出口発見率      92.0%   発見後クリア率 100%
  平均カバレッジ  85.5%   クリアまで平均164步
```

**アブレーション (観測の一部をゼロ埋め) — 何を学んだかの測定:**

| 消した観測 | クリア率 |
|---|---|
| なし | 92% |
| 未探索チャネル (local+global の ch0) | 87% |
| 全体マップ 24×24 全部 | 87% |
| 軌跡 (訪問回数) | 89% |
| **局所既知マップ 15×15 全部** | **8%** |

方策が実際に使っているのは**局所既知マップだけ**。9×9 迷路は 15×15 の窓に
ほぼ全部収まるので、局所マップが実質全体マップとして機能し、「どこをまだ
見ていないか」(未探索チャネル) を使う必要がなかった。**「探索を学んだ」と
言うにはまだ早い。** 未探索チャネルと全体マップが本当に必要になるのは、
マップが局所窓からはみ出す maze11 以降 — そこで同じアブレーションを再測する。

決定的方策だと 65% に落ちる (記憶が観測にしかない MLP は、決定的にすると
同じ場所でループしやすい)。探索タスクの評価は確率的で行うこと。

### 学習結果: 第2段階 maze11 (11×11 迷路, maze9 から引き継ぎ)

**学習コスト**: 8M 步を **91分** (ノートPC 12コア・96環境・平均 1,461 行動/秒)。

学習曲線 (clear_rate の推移)。maze9 の重みを引き継いだので、初動から高い:

```
  1.1M步  83%   ← 引き継ぎ直後から maze9 の最終値相当
  2.2M步  88%
  4.3M步  90%
  6.0M步  95%   ← ここでほぼ飽和
  8.0M步  92%前後で収束
```

**評価** (未知シードの迷路 100 エピソード。学習に出たマップは1つもない):

| | 確率的 | 決定的 |
|---|---|---|
| クリア率 | **98%** | 95% |
| 出口発見率 | 93% | 88% |
| 発見後クリア率 | 100% | 100% |
| 平均カバレッジ | 78.5% | 75.8% |
| クリアまでの所要 | 平均192步 (ゲーム内 12.7秒) | 平均151步 (10.0秒) |

気づき:

- **クリア率 (98%) > 出口発見率 (93%)**: 5エピソードは出口を「見ずに」クリアした。
  E キーは視線判定なしで正面1.5タイルに届くため、壁の角越しに押せる (ゲーム仕様)
- **決定的でも 95%**: maze9 時点では決定的にすると 65% に崩れていた
  (ループにはまる)。追加の 8M 步でループ耐性が付いた
- 発見後クリア率は常に 100%: 「見つけたのに辿り着けない」は起きていない。
  既知マップを辿るナビゲーションは完成している

**アブレーション** (観測の一部をゼロ埋め、確率的・各100エピソード):

| 消した観測 | クリア率 | 出口発見率 | 平均エピソード長 |
|---|---|---|---|
| なし | 98% | 93% | 200步 |
| 未探索チャネル | **98% (無傷)** | 89% | 205步 |
| 全体マップ 24×24 | 96% | 88% | 206步 |
| 軌跡 (訪問回数) | 95% | 92% | 198步 |
| **局所既知マップ 15×15** | **38%** | 29% | 465步 |

11×11 もまだ 15×15 の局所窓にほぼ収まるので、局所マップだけで解けてしまう。
**「どこをまだ見ていないか」が本当に必要になるのは窓からはみ出す maze15 以降** —
そこでこの表を再測することが「探索を学んだ」の判定になる。

### 学習結果: 第3段階 maze15 (15×15・braid 0.15、maze11 から引き継ぎ 20M步)

VM (EPYC 32vCPU) で70分、平均 4,700 行動/秒。**75%前後で頭打ち、終盤に劣化**:

```
  3M步 70% → 9M步 76% → 15M步 75% → 終盤 58〜61% に劣化
  評価: ckpt 16M = 74%  /  final 20M = 66%  → 採用は ckpt 16M
```

**アブレーションの再測** (ckpt 16M) — maze9/11 との比較が本命の結果:

| 消した観測 | maze9 | maze11 | maze15 |
|---|---|---|---|
| 未探索チャネル | -5pt | ±0 | **74→59% (-15pt)** |
| 局所既知マップ | -84pt | -60pt | 74→5% (-69pt) |

**マップが局所窓からはみ出した maze15 で、ようやく未探索チャネルが使われ始めた。**
「探索の学習」は起きているが、まだ弱い。

**失敗の解剖**: 失敗26件のうち出口発見済みは0件 (発見後クリア率は100%)。
失敗時のカバレッジは57.7%で、900步を使い切っても4割が未踏 = **探索がループして
停滞している**。成功と失敗を分けるのはナビ能力ではなく「探索を続けられるか」。

### 対策: フロンティアへのポテンシャル整形 (Lv1.5)

「新タイル +0.02」は発見の瞬間にしか出ない。近場を舐め尽くした後、次の
フロンティア (未知との境界) まで既知の通路を歩く区間は**報酬の砂漠**になり、
そこで方策がループする。そこで:

```
目標 (出口/キー) 未発見の間:
  整形報酬 = (前ステップのフロンティアまでの既知マップBFS距離 − 今の距離) × 0.05
目標発見後: 従来どおり目標への接近 × 0.1
```

- フロンティア = 未知の4近傍を持つ既知の床。**自分が見たタイルの地図から
  計算するのでカンニングではない** (「来た道は覚えている」の内側)
- 差分は「前ステップ時点の知識で張った場」で取る。発見でフロンティアが
  移動しても遡って罰しない
- 観測は Box(5866) のまま不変。既存チェックポイントから `--init` できる
- ただしこれは「どこへ探索に行くか」の判断を環境が肩代わりする Lv1.5 であり、
  Lv2 (自力探索) へ進むときはこの整形を外して LSTM に置き換えるのが筋

同条件 (maze11 の重みから 20M步) で再学習し、74% からの改善幅で効果を判定する。

**A/B の結果 (各100エピソード・未知シード): 整形の勝ち。**

| maze15 のモデル | クリア率 |
|---|---|
| 整形なし・ベスト (16M步) | 74% |
| **整形あり・6M步** | **89%** |
| 整形あり・final (20M步) | 76% |

整形なしが16M步かけて到達した性能を、**整形ありは6M步で15pt上回った**。
探索停滞 (報酬の砂漠) の診断は正しかった。

### 発見: 終盤の劣化はエントロピーの膨張

どちらの走行でも「後半にじわじわ劣化」が再現した (89→76% / 75→61%)。
TensorBoard の時系列を見ると、**方策のエントロピーが単調に膨らみ続け、
clear_rate の低下と同時進行**していた (整形なし: 5.1→6.2 nat)。

原因は力関係の逆転: `VecNormalize(norm_reward=True)` は報酬をリターンの走行
標準偏差で割るので、**方策が上達するほど勾配の信号が縮む**。一方エントロピー
ボーナス (`ent_coef=0.01`) は固定。長丁場では「ランダムでいろ」の圧力が勝ち、
方策がぼやける。KL・clip_fraction は安定していたので、大きな更新で壊れた
のではなく、目的関数のバランスが徐々に傾いた劣化。

対策: `EntCoefAnneal` (0.01 → 0.001 線形減衰) を train2.py に追加。
運用面では**ベストチェックポイント採用** (final を信用しない) を続ける。

### 次の段階: maze21-rooms への移行前チェック (教訓6)

maze15 チャンピオン (整形あり 6M) を maze21-rooms にそのままかけると **27%**。
0% ではないので、この段階は「少しだけ難しい」の範囲に収まっている。移行してよい。

## 付録: なぜ現状の設計を選んだのか(弁明)

「最初から探索版にすればよかったのでは」と思うかもしれません。実際そうかもしれません。
ただ、当時の判断は以下でした。

- ゴールが **「全5ステージのクリア」** だった(「人間らしいプレイの再現」ではない)
- 疎報酬問題を回避するため、BFS勾配が**最も確実な手**だった
- **CPUのみ**の環境で、現実的な時間(数時間)で結果を出す必要があった

そして結果的に、**この設計だからこそ発見できたことが多くありました**
(γの罠、報酬の非対称、破滅的忘却、解けないタスクの害、成功の誤解)。
**BFSオラクルという「簡単モード」で足場を固めたからこそ、次の難しい問題に進める**とも
言えます。

ただし、**「AIがDOOMをクリアした」と言うときは、必ず「与えられた経路を辿って」を
付け加えてください。** それが誠実さです。
