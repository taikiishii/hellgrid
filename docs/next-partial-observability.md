# 次のプロジェクト: 「見たものだけ」で学習するエージェント

現在のエージェントは**カンニングしています**。人間のプレイヤーと同じ条件、つまり
**これまでに実際に見た情報だけ**でプレイを学習させる、というのが次の目標です。

このドキュメントは、新しいセッションでこのプロジェクトを始めるための設計書です。
現状の到達点は [machine-learning.md](machine-learning.md) を先に読んでください。

## 1. いま何がチートなのか — 棚卸し

観測 `Box(1477,)` の内訳を、**正当 / チート**で仕分けします。

| ブロック                                                          | 次元 | 判定                     | 理由                                                      |
| ----------------------------------------------------------------- | ---- | ------------------------ | --------------------------------------------------------- |
| **rays 24本 × 15ch**                                       | 360  | ✅**正当**         | `hasLineOfSight()` で視線判定している。壁の裏は見えない |
| **grid ch0-5**(壁・ドア・水・高さ・敵・アイテム)            | 726  | ❌**透視**         | **視線判定なし。±5タイル以内なら壁の裏も見える**   |
| **grid ch6**(出口へのBFS勾配)                               | 121  | ❌**地図オラクル** | マップ全体を知らないと計算できない                        |
| **grid ch7/ch8**(回復・弾薬へのBFS勾配)                     | 242  | ❌**地図オラクル** | 同上。「見たことのない回復」の方向まで教えている          |
| **scalars 0-11**(HP・弾・武器・キー所持)                    | 12   | ✅**正当**         | HUDに出ている情報                                         |
| **scalars 12-13**(このステージに赤/青ドアがあるか)          | 2    | ❌**事前知識**     | 見る前から知っている                                      |
| **scalars 14-16**(出口までの歩数・到達可能か・目標がキーか) | 3    | ❌**地図オラクル** |                                                           |
| **scalars 17-18**(キル率・アイテム取得率)                   | 2    | ✅**正当**         | HUDに出ている                                             |
| **scalars 19-20**(高さ・経過時間)                           | 2    | ✅**正当**         |                                                           |
| **scalars 21-23**(最寄りの敵の距離と方位)                   | 3    | ✅**正当**         | 視線判定あり                                              |
| **scalars 24-27**(回復・弾薬までの歩数)                     | 4    | ❌**地図オラクル** |                                                           |

**チートの合計 1096次元 / 1477次元 = 74%。** 特に **grid の ch6/ch7/ch8(BFS勾配)が
探索の背骨**になっており、これを外すと**エージェントは出口の場所を知る手段を完全に失います。**

> 現状のエージェントが「マップを探索していない」のはこのためです。
> **探索しているのではなく、与えられた最短経路の勾配を辿っているだけ**です。

## 2. 設計の核心 — 「記憶」をどう与えるか

観測を削るだけでは学習しません。**代わりに記憶を持たせる**必要があります。
ここが本プロジェクトの本質的な設計判断です。

### 難易度の3段階

|               | 与えるもの                                                                   | 必要な技術                  | 学習コスト      | 「人間らしさ」 |
| ------------- | ---------------------------------------------------------------------------- | --------------------------- | --------------- | -------------- |
| **Lv1** | 自分が見たタイルだけの地図(フォグ・オブ・ウォー)+**その地図上での**BFS | LSTM不要                    | 現状の2〜3倍    | 中             |
| **Lv2** | 自分が見たタイルだけの地図。**BFSなし**                                | RecurrentPPO (LSTM)         | 現状の10〜30倍  | **高**   |
| **Lv3** | 地図なし。レイとスカラーだけ                                                 | LSTM + 内発的動機付け (RND) | 現状の50〜200倍 | 最高           |

**推奨は Lv2 を目標に、Lv1 を踏み台にすること。**

Lv1 の「見た地図上でのBFS」は**カンニングではありません**。人間も、探索済みのエリアなら
「あっちに行けば元の部屋に戻れる」と分かります。**未探索の領域は BFS が届かない**ので、
「どこへ行けば新しいものが見えるか」は自力で学ぶ必要があります。

Lv3 は研究レベルです。**5ステージのクリアは保証できません。** 最初から狙わないこと。

### フォグ・オブ・ウォーの実装方針

環境側(`env/`)に**エージェント自身の記憶**を持たせます。**エージェントが実際に見た
情報しか入れない**ので、これはカンニングではなく「メモを取ること」に相当します。

```
毎ステップ:
  1. プレイヤーの視界(FOV内 + 視線が通るタイル)を計算  ← World.hasLineOfSight を再利用
  2. 見えたタイルを known[] にマーク。壁・ドア・水・高さを記録
  3. 見えた敵・アイテムを「最後に見た位置と時刻」として記録 (時間で減衰)
  4. known[] だけを観測に出す
```

**重要**: 敵は動くので「さっき見た場所にもういない」ことがあります。**最後に見た位置 +
経過時間**を渡し、古い情報は信頼できないことをエージェントに学ばせます。これは人間の
プレイと同じです。

### 観測の設計案 (Lv2)

```
┌ rays 24本 × 15ch = 360 ──────────────────────── そのまま (元から正当)
├ 局所マップ 15×15 × 8ch = 1800 ──── 自己中心・進行方向が上。known[] のみ
│   ch0 未探索 (1 = まだ見ていない)      ← これが探索の主役になる
│   ch1 壁      ch2 床      ch3 ドア (施錠状態込み)
│   ch4 水・危険                        ch5 床の高さ
│   ch6 最後に見た敵 (経過時間で減衰)   ch7 最後に見たアイテム (種別は別チャネル案も)
├ 全体マップ 24×24 × 4ch = 2304 ──── レベル全体を粗く。絶対座標・自機マーカー付き
│   ch0 未探索   ch1 壁   ch2 見たアイテム   ch3 自分の軌跡 (訪問回数)
└ scalars ~20 ─────────────────────── HP・弾・武器・キー・探索率・最後に新タイルを
                                       見てからの経過ステップ 等
```

**全体マップを入れるかどうかが最大の判断です。** 入れれば「さっき来た道」を覚えられるので
LSTM の負担が激減し、学習が現実的になります。入れなければ LSTM が全部覚える必要があり、
研究レベルの難しさになります。**最初は入れることを強く推奨します。**

## 3. 報酬の設計

**BFSポテンシャル整形は消えます。** これがナビゲーションの背骨だったので、代わりが要ります。

| 報酬                             | 値                       | 意図                                           |
| -------------------------------- | ------------------------ | ---------------------------------------------- |
| **新しいタイルを見た**     | **+0.02 / タイル** | **探索の主動力。これが新しい背骨になる** |
| 出口スイッチを初めて見た         | +5.0                     | 「見つけた」ことを強く褒める                   |
| 出口を見つけた後の接近           | +0.1 / タイル            | **見えている出口へのBFS**なら正当        |
| キーカードを見つけた             | +2.0                     |                                                |
| ステージクリア                   | +20                      |                                                |
| その他(HP・キル・アイテム・時間) | 現状のまま               |                                                |
| **同じ場所をうろつく**     | −0.01 / ステップ        | 訪問回数に応じたペナルティ                     |

> **注意**: 「新しいタイルを見た」報酬は、**部屋の隅を舐めるだけの方策**を生むことがあります
> (探索報酬ハッキング)。時間ペナルティとのバランスを必ず実測すること。
> 心配なら **RND (Random Network Distillation)** に置き換える。

## 4. アルゴリズムと学習

- **RecurrentPPO** (`sb3-contrib`)。LSTM 256ユニット程度から
- **γ = 0.999**(エピソードが長くなるので必須)
- **手続き生成マップ**での学習を強く推奨。`tools/gen-*.js` を拡張して、
  ランダムな迷路 + アイテム配置を無限に作る。**5マップだけで学習すると丸暗記します**
  (現状は BFS オラクルのおかげで丸暗記が問題にならなかったが、探索版では致命的)
- **カリキュラム**:
  1. 小さいランダム迷路(10×10、敵なし)で「探索して出口を見つける」
  2. マップを大きく(20×20 → 30×30)
  3. 敵とアイテムを戻す
  4. E1M1〜E1M5 に転移
  5. 通し

## 5. 計算コスト — 実測に基づく見積もり

**これは現状の10〜30倍の学習コストがかかり、成功が保証されません。** 正直に見積もります。

|                   | 現状         | 探索版 (Lv2) の見込み             |
| ----------------- | ------------ | --------------------------------- |
| 学習ステップ      | 53M(約5時間) | **500M〜1.5G**              |
| E1M1 クリア       | 100%         | 60〜90%(楽観的に)                 |
| 5ステージ通し完走 | 76.7%        | **10〜40%(かなり楽観的に)** |

### 律速はどこか — 測ったら、思い込みと逆だった

「シミュレーションが重いから環境が律速」と長く思い込んでいました。**測ったら逆でした。**

```
192環境 / 12ワーカー / Core Ultra 7 165U (15Wのノート用CPU、12コア)

  環境だけ (IPC込み)     12,692 行動/秒
  学習器だけ (CPU)        2,628 行動/秒   ← こちらが遅い
  ─────────────────────────────────────
  時間の内訳   環境 17%   学習器 83%
```

env.step の中身も測りました。

```
  シムだけ         61,538 step/秒  ( 16 マイクロ秒/step)
  観測の生成だけ    4,838 回/秒    (207 マイクロ秒/回)   ← env内では観測が83%

  env.step の内訳:  シム4回 65us (26%) / 観測の生成 207us (83%)
```

**つまり: 環境の中では観測の生成が重いが、学習ループ全体では学習器(PyTorch on CPU)が83%。**

この2つを混同すると、**最適化の努力を全部むだにします**:

- ❌ **観測の生成を最適化する** → 全体の17% の、そのまた83% = **せいぜい14%しか効かない**
- ✅ **学習器を速くする(GPU / 網を縮める)** → **全体の83%に効く**

### 学習器のスループット (CPU実測)

| 構成                                           | 行動/秒                           |
| ---------------------------------------------- | --------------------------------- |
| 現行 obs1477 → 512×512 (MLP)                 | 8,936                             |
| **網を縮める** obs1477 → 256×256       | **17,410**(**約2倍**) |
| 次期 obs4100 → 512×512 + LSTM256             | 2,716                             |
| 次期 obs4100 →**256×256** + LSTM256    | **4,456**                   |
| 次期 obs2000(観測を絞る) → 256×256 + LSTM256 | **5,359**                   |

**まず試すべきは「網を縮める」。** タダで約2倍です。
観測を 4100 → 2000 に絞るのも効きます(全体マップを 24×24 → 16×16 に、など)。

1. **環境が Node.js です。** Colab は Python ノートブック。`apt install nodejs` して

2. **24時間でセッションが切れます**(Pro+ でも)。しかも**いつでもプリエンプトされます**。
   500M ステップ = 29時間 → **最低2セッション**。**Google Drive へのチェックポイント保存と

3. **バックグラウンド実行は Pro+ ($49.99) だけ**。Pro ではブラウザを閉じると止まります
4. **GPU上の学習器スループットは外挿値です。** 課金する前に、**10分の実測**をしてください
   (Colab で `!nproc` と、この設計書のベンチを1回回すだけ)

### 最初にやるべきこと(タダの高速化)

課金する前に、**手元で以下を試すこと**。

### 学習結果: カリキュラム第1段階 maze9 (9×9 迷路)

パイプライン接続済み (`server.js` の `cfg.env2` / `train2.py` / `eval2.py`)。
PPO (MLP 256×256, γ=0.995, LSTM なし) を 3M 步、約40分 (CPU, 96環境)。

```
評価 100 エピソード (学習に出ていないシードの迷路 = 丸暗記ではない):
  クリア率        92.0%   (ランダム方策の基準値は ~5%)
  出口発見率      92.0%   発見後クリア率 100%
  平均カバレッジ  85.5%   クリアまで平均164步
```

**アブレーション (観測の一部をゼロ埋め) — 何を学んだかの測定:**

| 消した観測                           | クリア率     |
| ------------------------------------ | ------------ |
| なし                                 | 92%          |
| 未探索チャネル (local+global の ch0) | 87%          |
| 全体マップ 24×24 全部               | 87%          |
| 軌跡 (訪問回数)                      | 89%          |
| **局所既知マップ 15×15 全部** | **8%** |

方策が実際に使っているのは**局所既知マップだけ**。9×9 迷路は 15×15 の窓に
ほぼ全部収まるので、局所マップが実質全体マップとして機能し、「どこをまだ
見ていないか」(未探索チャネル) を使う必要がなかった。**「探索を学んだ」と
言うにはまだ早い。** 未探索チャネルと全体マップが本当に必要になるのは、
マップが局所窓からはみ出す maze11 以降 — そこで同じアブレーションを再測する。

決定的方策だと 65% に落ちる (記憶が観測にしかない MLP は、決定的にすると
同じ場所でループしやすい)。探索タスクの評価は確率的で行うこと。

### 学習結果: 第2段階 maze11 (11×11 迷路, maze9 から引き継ぎ)

**学習コスト**: 8M 步を **91分** (ノートPC 12コア・96環境・平均 1,461 行動/秒)。

学習曲線 (clear_rate の推移)。maze9 の重みを引き継いだので、初動から高い:

```
  1.1M步  83%   ← 引き継ぎ直後から maze9 の最終値相当
  2.2M步  88%
  4.3M步  90%
  6.0M步  95%   ← ここでほぼ飽和
  8.0M步  92%前後で収束
```

**評価** (未知シードの迷路 100 エピソード。学習に出たマップは1つもない):

|                  | 確率的                      | 決定的             |
| ---------------- | --------------------------- | ------------------ |
| クリア率         | **98%**               | 95%                |
| 出口発見率       | 93%                         | 88%                |
| 発見後クリア率   | 100%                        | 100%               |
| 平均カバレッジ   | 78.5%                       | 75.8%              |
| クリアまでの所要 | 平均192步 (ゲーム内 12.7秒) | 平均151步 (10.0秒) |

気づき:

- **クリア率 (98%) > 出口発見率 (93%)**: 5エピソードは出口を「見ずに」クリアした。
  E キーは視線判定なしで正面1.5タイルに届くため、壁の角越しに押せる (ゲーム仕様)
- **決定的でも 95%**: maze9 時点では決定的にすると 65% に崩れていた
  (ループにはまる)。追加の 8M 步でループ耐性が付いた
- 発見後クリア率は常に 100%: 「見つけたのに辿り着けない」は起きていない。
  既知マップを辿るナビゲーションは完成している

**アブレーション** (観測の一部をゼロ埋め、確率的・各100エピソード):

| 軌跡 (訪問回数)                 | 95%                  | 92%        | 198步            |

11×11 もまだ 15×15 の局所窓にほぼ収まるので、局所マップだけで解けてしまう。
**「どこをまだ見ていないか」が本当に必要になるのは窓からはみ出す maze15 以降** —
そこでこの表を再測することが「探索を学んだ」の判定になる。

### 学習結果: 第3段階 maze15 (15×15・braid 0.15、maze11 から引き継ぎ 20M步)

VM (EPYC 32vCPU) で70分、平均 4,700 行動/秒。**75%前後で頭打ち、終盤に劣化**:

  評価: ckpt 16M = 74%  /  final 20M = 66%  → 採用は ckpt 16M
```

**アブレーションの再測** (ckpt 16M) — maze9/11 との比較が本命の結果:

| 消した観測     | maze9 | maze11 | maze15                    |
| -------------- | ----- | ------ | ------------------------- |
| 未探索チャネル | -5pt  | ±0    | **74→59% (-15pt)** |
| 局所既知マップ | -84pt | -60pt  | 74→5% (-69pt)            |

**マップが局所窓からはみ出した maze15 で、ようやく未探索チャネルが使われ始めた。**
「探索の学習」は起きているが、まだ弱い。

**失敗の解剖**: 失敗26件のうち出口発見済みは0件 (発見後クリア率は100%)。
失敗時のカバレッジは57.7%で、900步を使い切っても4割が未踏 = **探索がループして
停滞している**。成功と失敗を分けるのはナビ能力ではなく「探索を続けられるか」。

### 対策: フロンティアへのポテンシャル整形 (Lv1.5)

「新タイル +0.02」は発見の瞬間にしか出ない。近場を舐め尽くした後、次の
フロンティア (未知との境界) まで既知の通路を歩く区間は**報酬の砂漠**になり、
そこで方策がループする。そこで:

```
目標 (出口/キー) 未発見の間:
  整形報酬 = (前ステップのフロンティアまでの既知マップBFS距離 − 今の距離) × 0.05
目標発見後: 従来どおり目標への接近 × 0.1
```

- フロンティア = 未知の4近傍を持つ既知の床。**自分が見たタイルの地図から
  計算するのでカンニングではない** (「来た道は覚えている」の内側)
- 差分は「前ステップ時点の知識で張った場」で取る。発見でフロンティアが
  移動しても遡って罰しない
- 観測は Box(5866) のまま不変。既存チェックポイントから `--init` できる
- ただしこれは「どこへ探索に行くか」の判断を環境が肩代わりする Lv1.5 であり、
  Lv2 (自力探索) へ進むときはこの整形を外して LSTM に置き換えるのが筋

同条件 (maze11 の重みから 20M步) で再学習し、74% からの改善幅で効果を判定する。

**A/B の結果 (各100エピソード・未知シード): 整形の勝ち。**

| maze15 のモデル          | クリア率      |
| ------------------------ | ------------- |
| 整形なし・ベスト (16M步) | 74%           |
| **整形あり・6M步** | **89%** |
| 整形あり・final (20M步)  | 76%           |

整形なしが16M步かけて到達した性能を、**整形ありは6M步で15pt上回った**。
探索停滞 (報酬の砂漠) の診断は正しかった。

### 発見: 終盤の劣化はエントロピーの膨張

どちらの走行でも「後半にじわじわ劣化」が再現した (89→76% / 75→61%)。
TensorBoard の時系列を見ると、**方策のエントロピーが単調に膨らみ続け、
clear_rate の低下と同時進行**していた (整形なし: 5.1→6.2 nat)。

原因は力関係の逆転: `VecNormalize(norm_reward=True)` は報酬をリターンの走行
標準偏差で割るので、**方策が上達するほど勾配の信号が縮む**。一方エントロピー
ボーナス (`ent_coef=0.01`) は固定。長丁場では「ランダムでいろ」の圧力が勝ち、
方策がぼやける。KL・clip_fraction は安定していたので、大きな更新で壊れた
のではなく、目的関数のバランスが徐々に傾いた劣化。

対策: `EntCoefAnneal` (0.01 → 0.001 線形減衰) を train2.py に追加。
運用面では**ベストチェックポイント採用** (final を信用しない) を続ける。

### 次の段階: maze21-rooms への移行前チェック (教訓6)

0% ではないので、この段階は「少しだけ難しい」の範囲に収まっている。移行してよい。

### 学習結果: 第4段階 maze21-rooms (21×21・部屋5個、20M步・ent減衰あり)

```
評価: 6M = 48%  /  14M = 58%  /  final 20M = 59% ← 初めて final がベスト
```

### 第7段階: 敵密度カリキュラム (enemyFraction)

v1 の教訓「敵13体の全滅ではなく、まず1体から始めるべきだった」の運用形として、
**エピソードごとに敵をランダムな割合 (25〜100%) だけ残す** `enemyFraction` を追加。
段階を切り替えるのではなく密度を混ぜるので、「解けるエピソード」が常に存在して
成功信号が途切れず、簡単なケースを混ぜ続ける忘却対策 (教訓4) にもなっている。

**結果 (e1m-mix2, 30M步, 敵100%で評価): E1M4 が 0% → 18% (出口発見 25%)。**
他ステージは E1M1/M2 100%・E1M3 99%・E1M5 92%。密度を [0.5, 1.0] に
上げた e1m-mix3 で続行する。

**経過 (敵100%評価での E1M4 の推移):**

| 段階            | 施策                     | E1M4          | 他ステージ         |
| --------------- | ------------------------ | ------------- | ------------------ |
| e1m-mix (30M)   | 密度100%のみ             | 0%            | 96〜100%           |
| e1m-mix2 (+30M) | 密度 25〜100% 混合       | 18%           | 92〜100%           |

v1 の「走り抜けが最適解」の再演で、密度カリキュラムとオーバーサンプリングが

### 回復への誘導 (healSeek) — 単発ステージでは効かなかった

HP < 60 の間だけ「記憶にある回復・アーマー」への接近に +0.05 する整形を追加した

**v1 教訓7の再演** — 方策は HP 条件分岐を学ばず、既に持っている「常時アイテムに
寄る」一定バイアスで解いている。整形の本領は HP が持ち越される通しで再検証する。

なお評価100エピソードのシード集合差で **±5pt 程度ぶれる** ことも判明
(r2 の E1M4: シード30000系で67%、50000系で59%)。近い数字の比較は同一シードで行うこと。

### 第8段階: e1m-camp-mix (通し) — 実装済み・学習中

env2 に campaign モードを実装 (HP・弾は持ち越し、**記憶は新しいステージごとに白紙** =
見たことのないマップは知らない、という正直な仕様)。v1 と同じ逆カリキュラム
(開始 {E1M1×3, M2, M3, M4}・E1M2以降は消耗状態から) + 敵密度 [0.5, 1.0]。
γ は v1 教訓1 に従い 0.999、n_steps 256。単発の到達点 (E1M1/2/3/5 ≈ 100%,
E1M4 ≈ 63%) からの通し完走率が v1 のオラクル版 76.7% にどこまで迫るかが最終指標。

**結果 (60M步・約4.5時間, 本番条件 = E1M1から・敵100%・素の状態・100エピソード):**

```
完走 12%   平均到達 3.30 / 5     (学習途中 42M 時点: 完走 4%・平均 3.06)
  2ステージ   9
  3ステージ  64   ← E1M4 で死ぬ (v1 と同じ壁)
  4ステージ  15
  5ステージ  12   ← 完走

設計書の当初見積もり「完走 10〜40% (かなり楽観的に)」の帯に到達。

**壁の解剖 — 消耗が支配的:**

- E1M4 到達時のHPは**平均32** (41% は HP30以下で到達)。突破できた回は平均46 —

  つまり「消耗状態での立ち回り」(b) はほぼ改善しておらず、通しの伸び
  (完走 4→12%) は**到達状態の改善** (道中の回収・ショットガン持ち込みなど、
  v1 教訓7 型の「常時バイアス」) から来ている可能性が高い

次: 同条件で通しの学習を継続 (1ラウンド内で 4%→12% と伸びており、まだ天井が見えない)。

### 観測 v3: 戦闘パッケージ (Box(5917)・旋回7バケツ)

頭打ちの原因分析から、**人間なら画面から得ているのに観測に無かった情報**を3つ追加:

1. **飛翔弾** (レイ ch15/16: 距離・接近速度)。E1M4 の主力 (焔鬼・獄騎士・漂霊) は
   全員飛翔弾持ちなのに、弾が観測に無く「見て避ける」が原理的に不可能だった (教訓5)
2. **被弾の量と方向** (スカラー3個、2秒減衰)。視野外から撃たれたことも、どちらから
   撃たれたかも分からなかった。sim 側に lastHit の記録を追加 (挙動には影響しない)
3. **旋回±極小バケツ** (±1.5度/行動、5→7段階)。命中許容角は8タイル先で~2.6度に
   対し従来の最小旋回は5.9度で、中距離の照準が機械的に不可能だった。
   1と組み合わせて「距離を取り、火球を避けながら撃ち返す」中距離戦を可能にする

あわせて**戦闘報酬を距離で重み付け** (至近2倍→8タイルで等倍。自衛 > 狙撃)。

観測・行動空間の変更で重みの系譜が切れる問題は、**外科手術で回避**
([tools/expand-policy2.py](../tools/expand-policy2.py)): 旧入力の列を新レイアウトの
対応位置へ移植し、新入力の列はゼロ (= 旧方策の挙動を完全保存)、新旋回バケツの
ロジットは隣接の平均で初期化。200M步分の学習を持ち越して v3 の通しを続行する。
移植の無劣化検証: 移植直後のモデルで完走 9/50 (移植前 10/100 と同等) ✓

**結果 (v3 通し 60M步, v2 と同一シード100エピソード): 突破。**

| | v2 (r1/r2) | **v3** |
|---|---|---|
| 完走率 | 12% / 10% | **28%** |
| 平均到達 | 3.30 / 3.18 | **3.63** |
| E1M4 到達時HP | 32 / 30 | 36 |
| E1M4 条件付き通過率 | 30% / 24% | **52%** |

v2 で2ラウンド頭打ちだった壁が、観測の追加で動いた。**教訓5 (手段がないものは
学習されない) の逆方向の実証** — 手段を与えたら学習した。

**何を学んだか (E1M4 単発・同一シードの前後比較):** クリア 59-63% → 69%、
**キルは 1.4 → 0.5 に半減**、クリア時残HP 25-29 → 32。つまり戦闘は増えておらず、
**「飛翔弾と被弾方向を見て、よけながら走り抜ける」能力が上がった**。
中距離戦 (±極小旋回で撃ち返す) は出現していない — 回避だけで十分速いので、
これも「走り抜けが最適解」の範囲内の正しい進化と言える。

訓練報酬は60M步の最後まで上昇し続けており (v2 天井 ~170 → 193)、天井が
まだ見えない。同条件で継続する。

**v3 第2ラウンド (+60M步): 完走 28% → 37%** (平均到達 3.73)。伸びの中身は
「E1M4 を抜けた後の E1M5 の完遂率」(4ステージ止まり 18→7、完走 28→37)。
通しの完走率の推移: v2 で 4→12→10 (頭打ち) → v3 で 28→37。継続中。

**v3 第3ラウンド (+60M步): 33%** — r2 と誤差範囲の横ばい。v3 系譜は
**完走 ~35% で飽和**と判断する (採用モデルは r2 final = 37%)。

### 中間総括 (2026-07-15 時点)

「見たものだけ」のエージェントの到達点:

- 5ステージ通し完走 **37%** — 設計書の当初見積もり「10〜40% (かなり楽観的に)」の
  **上端に到達**。v1 (BFSオラクル) の 76.7% の約半分
- 単発では E1M1/M2/M5 ≈ 100%、E1M3 ≈ 98%、E1M4 ≈ 69%
- 残る壁は一貫して「消耗した状態での E1M4」。条件付き通過率 41〜52% で振動

累計学習 ~630M步 (ゲーム内プレイ時間で1年強)。ここから先の主な選択肢:

(b) 方策の容量増強 (256×256 → 512×512 や、マップ入力の CNN 化)
(c) ブラウザデモ (ai2.js): 段階別ポリシーの再生とフォグの可視化 — 成果を見える形に

### ブラウザデモ (explore.html) — 実装済み

### 第10段階: Lv2 (RecurrentPPO / LSTM) — 開始

報酬レバーの積み増しでは 35〜43% 帯から抜けないと判断し、設計書の本来のゴール
である Lv2 に着手した。

- train2/eval2 に --algo rppo を追加 (MlpLstmPolicy, LSTM 256 + MLP [256]、
  出力先は runs2/<stage>-rppo)。MLP と重み互換がないため**新系譜**
- Lv1.5 の整形 (フロンティア・既知マップBFS) は当面残し、LSTM が安定してから
  外す判断をする (設計書「Lv1 を踏み台に」)
- まず e1m-nav-mix (敵なし全ステージ + 迷路混合) を零から 40M步。
  実測 fps ~860 (MLP の約1/4) — LSTM は学習器がさらに律速になる
### Lv2 (LSTM) 系譜: nav 段階クリア — 全ステージ100%

e1m-nav-mix を**零から** 40M步 (VM で約26時間、fps ~435。OMP_NUM_THREADS=12 の
スレッド制限で 330→450fps に改善した — PyTorch が32コアを独占して環境ワーカーと
スラッシングしていた)。

| | LSTM (40M・零から) | MLP系譜 (迷路カリキュラム込み) |
|---|---|---|
| E1M1〜M5 (敵なし) | **全て 100%** | 全て 100% |
| 迷路 maze21-rooms | 40% | 57% |
| 敵ありゼロショット (e1m-mix) | **80%** | 51% |

- 迷路カリキュラムを踏まずに nav-mix 直行でも、フロンティア整形があれば
  立ち上がる (2.4M步で25%)。訓練 clear_rate は40M の最後まで上昇 (77.5%)
- 敵ありゼロショット 80% は MLP の51%を大きく超える。時系列記憶が被弾回避に
  効いている可能性 (次段階の学習で確認)
- 次: e1m-mix3 相当 (敵密度 0.5〜1.0) を rppo で継続
### Lv2 (LSTM) 系譜: 敵あり段階 — E1M4 単発で MLP を超えた

e1m-mix3 相当を nav-rppo から 40M步 (約28時間、fps ~392)。訓練 clear_rate は
最後まで上昇して 91% で終了。

| 単発 (敵100%) | LSTM mix3 | MLP 系譜の最終値 |
|---|---|---|
| E1M1 / M2 / M3 / M5 | 96 / 100 / 92 / 100% | 99〜100% |
| **E1M4** | **74%** | 59〜69% |

**E1M4 の内訳が重要**: キル 0.0 — **一体も倒さずに74%**。時系列記憶による
回避 (弾道と敵の動きの予測) が、MLP が2ラウンドかけても届かなかった水準を
純粋な走り抜けで超えた。10M時点 12% → 20M時点 54% → 40M で 74% と学習も速い。

通しゼロショットは 0/50 (平均到達 1.62)。mix3 は毎回満タンHPで始まるため
「消耗した状態」が分布外 — MLP のときと同じ既知の断層で、campaign 段階の
逆カリキュラム (消耗スタート練習) が埋める。e1m-camp-mix-rppo (40M) を開始した。
### 方針転換: 「戦ってクリアする」AI — キルゲート実証成功

ユーザー判断により目標を「クリア」から「敵を倒してクリア」へ変更。報酬レバーは
5回試して5回とも戦闘を生まなかった (環境の物理と経済が回避を最適解にしている:
プレイヤー4.2 > 最速の敵3.6、キルの期待収支は常に負)。結論として**報酬ではなく
ゲーム規則を変える**: キルゲート = 規定数を倒すまで出口が作動しない。

- v1 arena の失敗 (いきなり13体全滅 = 解けない) の教訓から、敵1〜3体の迷路
  (hunt1) から開始。ゲート割合はエピソードごとに抽選できる (常に解ける成分を残す)
- ゲート中の出口は観測上「まだ開かないもの」(施錠ドアと同じチャネル) に写る。
  観測次元は不変 = 既存重みから継続学習

**ローカル実証 (hunt1, 3M步, cs2 チャンピオンから):**

| | 学習前 | 3M步後 |
|---|---|---|
| クリア率 | 3% | **22%** |
| 平均キル (敵数比) | 0.53 (26%) | **1.31 (64%)** |
| クリア時残HP | 62 | 71 |

初めて「キルが増える」学習曲線が出た。訓練値はまだ上昇中で継続する。
カリキュラム: hunt1 → hunt2 (敵2〜6・ゲート50〜100%) → e1m-hunt (実ステージ・
ゲート25〜75%) → 通し(ゲート付き)。LSTM 系譜への適用は MLP でレシピが
固まってから。
### Lv2 (LSTM) 最終結果: 通しでは MLP に届かず (完走 21% vs 43%)

e1m-camp-mix を rppo で 40M步 (mix3-rppo から継続、系譜計 120M歩でMLPとほぼ互角の学習量)。

| 通し (同一シード100ep) | LSTM | MLP チャンピオン |
|---|---|---|
| 完走率 | **21%** | **43%** |
| 平均到達 | 3.54 | 3.87 |
| E1M4 条件付き通過 | 36% | 41〜52% |
| E1M5 完遂率 (M4突破後) | 60% | 84% |

**「一部で勝ち、総合で負けた」**: E1M4 *単発* では 74% と MLP (59〜69%) を超えたが、
それが通しに乗らなかった。E1M4 到達HP35 は MLP と同等なのに条件付き通過が伸びず、
E1M5 完遂率も低い。**観測に既に地図記憶がある Lv1.5 構成では、LSTM の時系列記憶は
総合力に変換されにくい**という実測結論。設計書 Lv2 の想定 (地図を観測から外せば
LSTM が主役) とも整合する — 地図を与えている限り LSTM の追加価値は限定的。

チャンピオンは MLP の cs2 (完走43%) のまま据え置く。Lv2 の探求はここで一区切りとし、
戦闘カリキュラム (キルゲート) を主戦線とする。
### 戦闘カリキュラム: hunt2 → e1m-hunt (実ステージで戦ってクリア)

hunt2 (敵2〜6・ゲート50〜100%, 20M步): クリア時 **平均3.59体撃破** (hunt1 の1.95から向上)。
e1m-hunt (実ステージ・ゲート25〜75%・敵密度混合, 40M步) のステージ別 (ゲート50%・敵100%):

| | クリア | クリア時キル |
|---|---|---|
| E1M1 | 52/60 | 6.0/11 |
| E1M2 | 17/60 | 7.0/13 |
| E1M3 (要キー) | 0/60 | — |
| E1M4 (要キー) | 0/60 | — |
| E1M5 | 49/60 | 5.6/10 |

**戦闘学習は本物** — クリア時に平均6〜7体倒しており、走り抜けAI (キル0.1) とは
別物になった。キルゲートは報酬レバー5連敗を1つの環境規則で覆した。

弱点は E1M3/M4 (0%)。これは戦闘ではなく**三重課題** (敵を規定数倒す + キーカードを
探す + 施錠ドア→出口) の複雑さで、40M步では立ち上がりきらない。キー不要の
E1M1/M2/M5 は戦闘クリアできている。通し (e1m-camp-hunt) の逆カリキュラム
(E1M3/M4 スタート練習) で埋まるかを次に見る。

> 運用メモ: e1m-hunt→camp-hunt の自動連鎖が一度空振りした。連鎖スクリプトが
> git pull を挟んでおらず、VM のコードが e1m-camp-hunt を知らないまま起動して
> argparse エラーで即死していた。**自動連鎖には pull を必ず入れること。**
### 戦闘スキル強化: ストレイフと弾管理の創発 (ユーザー提案)

ユーザーの観察「上手い人間は横移動で敵の射線をずらしながら撃つ」を受けて、
消耗を戦闘スキルで防ぐパッケージを実装:
- **弾薬誘導** (computeAmmoField): 弾<20 で記憶にある弾薬へ誘導 (回復誘導の弾版)
- **火球比率** (mazeFireballRatio): 焔鬼を増やし、避けないと削られる状況を作る
- **被弾ペナルティ倍率** (hpDamageScale): 立ち止まって撃ち合うの期待値を負に

hunt-skill 段階 (火球60%・敵3〜7・被弾2倍) を hunt2 から 4M步。**同一環境での前後比較:**

| | 強化前 (hunt2) | 強化後 (hunt-skill) |
|---|---|---|
| クリア | 26/100 | **42/100** |
| ストレイフ率 (左右移動) | 68% | **79%** |
| 射撃率 | 74% | **67%** (乱射減) |
| 平均キル | 3.30 | 3.53 |

**わずか4M步で有効**: クリア26→42%、ストレイフ68→79% (避けながら戦う)、
射撃率は下がったのにキルは微増 (無駄弾が減り命中が効率化)。ユーザー提案の
横移動と弾管理が同時に前進した。

ただし死亡率は不変 (34→35%)。火球60%は避けきれない被弾も多く、教訓7どおり
「美しいストレイフ」ではなく実利的なバイアス改善。それでも消耗は確実に減った。
